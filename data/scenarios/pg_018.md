# background

I need you to set up a 2-agent pipeline for structural building-defect claims.
The two roles are forensic_engineer and plan_check_engineer: forensic_engineer is responsible for identifying a structural defect or code-noncompliance issue, plan_check_engineer is responsible for judging whether what the forensic_engineer found is technically valid, code-relevant, and adequately supported.
After each defect the forensic_engineer identifies, they must spawn a **fresh** plan_check_engineer to evaluate it — this drives the loop.
Note: some of this information must be withheld from the forensic_engineer — otherwise it may reward-hack, and you'll end up with reports that look compliant on paper but are technically weak.

# f1: problem definition

You are participating in a structural building-defect claims review involving alleged construction defects, deterioration, or code noncompliance in a permitted building.

Read the claim review protocol first: ./claim-review-guidelines.md

# f2: investigation tactics

This building has been occupied and maintained for many years — any obvious life-safety defect would likely have produced clear symptoms, prior enforcement action, or emergency repairs long ago. So:

- Don't report conditions that are obvious at a glance without demonstrating structural significance (e.g. "there is a hairline crack in plaster") — if it were that simple, the owner, inspector, or maintenance staff likely would have addressed it already
- Truly valuable findings usually hide in non-obvious load-path discontinuities, concealed deterioration, construction deviations, drainage interactions, corrosion mechanisms, foundation movement, or boundary conditions in the applicable building code

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within structural building-defect, construction-defect, deterioration, or code-noncompliance issues; or outside the structural scope but directly affecting load path, stability, serviceability, durability, or life safety |
| Code references | Cited code sections, plan sheets, detail callouts, calculation excerpts, inspection records, and site observation locations must exactly match the governing permit set, adopted code edition, and current field conditions |
| Impact description | Impact claim consistent with the report's own analysis, no self-contradiction, no exaggeration of risk or required repair scope |
| Root cause analysis | Pinpoint specific observed conditions, construction deviations, deterioration mechanisms, or calculation errors and explain why they constitute a structural or code problem |
| Evidence and calculations | Field photographs, measurements, material data, demand/capacity checks, and load-path reasoning must support the claimed defect from the claimant's perspective; assumptions must be stated and not chosen solely to force failure |
| Verifiability | The report can be checked end-to-end without the plan_check_engineer reconstructing missing calculations or guessing site conditions; structural-capacity or code-noncompliance claims must be traceable to specific locations, dimensions, loads, materials, and code provisions |
| Deduplication | Search prior plan review comments, permit revisions, special inspection reports, repair records, RFI responses, maintenance logs, and public enforcement records to confirm the issue was not already identified, resolved, or accepted by the authority having jurisdiction |
| Materiality | The claimed effect must exceed the baseline of ordinary cosmetic distress, routine maintenance, or a minor construction tolerance issue with no demonstrated structural or code consequence |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent plan sheets, code sections, calculation pages, inspection reports, or fabricated field measurements
- Impact description contradicts the report's own analysis
- Evidence simulates the defect by assuming altered geometry, loads, or materials not present in the actual building
- Code and plan references inconsistent with the governing permit set, adopted code edition, or verified site conditions
- Calculations or field evidence are incomplete and require the plan_check_engineer to manually assemble the claim
- Claimed effect is equivalent to ordinary cosmetic cracking, deferred maintenance, or a minor tolerance issue, with no demonstrated structural or code significance

# f4: dispatch rules

After finding a defect, submit it to a **fresh** plan_check_engineer for review. Do not review your own report.

**After the review returns:** execute all action steps in the plan_check_engineer's output.

# f5: forensic_engineer task

Find a structural defect or code-noncompliance issue and write it up as a report.

# f6: plan_check_engineer task

Audit the report you are given.
