# background

I need you to set up a 3-agent pipeline for bridge inspection escalation: inspection scheduler, bridge inspection engineer, and structural peer reviewer.

- **inspection scheduler** orchestrates: calls bridge inspection engineer, extracts the defect report file path from bridge inspection engineer's reply, then calls structural peer reviewer on that path, decrements a counter, and loops. Does no engineering analysis itself.
- **bridge inspection engineer** reviews the bridge files and field observations, selects a defect or deterioration concern to investigate, and writes a structural defect report file.
- **structural peer reviewer** challenges the report against inspection escalation criteria and persists the verdict.

# f1: problem definition

You are participating in a bridge inspection escalation workflow for in-service highway bridges where potential structural defects must be screened for safety significance, code basis, severity rating, and repair urgency.

Read the escalation and inspection rules first: ./inspection-rules.md

# f2: inspection tactics

Most bridges in this program have been inspected on regular cycles for many years — any obvious severe defect would likely already be posted, repaired, or captured in prior inspection notes. So:

- Don't escalate conditions that are obvious at a glance and already routine maintenance items (e.g. "paint flaking on a non-structural railing") — if it were that straightforward, the owner would likely already have it in the maintenance queue
- Truly valuable escalations often hide in non-obvious interactions: fatigue-sensitive details, load-path changes after retrofits, bearing restraint, drainage-driven section loss, pack rust, fracture-critical members, scour exposure, or boundary conditions missed by standard element ratings

**Writing the report**

Each concern corresponds to its own report file at `reports/{id}-{slug}.md`, where `{id}` and `{slug}` come from the concern's id and slug attributes in ideas.xml.

The report must contain four h2 sections in this fixed order:

- `## Report Title` — concise, clear structural defect title
- `## Impact Description` — what damage or operational consequence it can cause, such as load posting, lane closure, accelerated fatigue, loss of redundancy, bearing seizure, substructure instability, or emergency repair
- `## Short Description` — short writeup of what's wrong, with bridge element, span, member, plan sheet, inspection photo, calculation, and code-clause references
- `## Reproduction Details` — enough field evidence and calculation steps for the structural peer reviewer to independently verify the concern as-is, including measurements, assumptions, load rating inputs, code checks, and any scripts or worksheets used

Do not modify or delete the `## Review` section — the structural peer reviewer overwrites that themselves.

If after investigation you find the concern doesn't hold, still briefly record your analysis and the reason for rejection in the report file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within bridge structural inspection, load rating, deterioration, fatigue, scour, bearing, deck, substructure, or repair-urgency issues; or outside those areas but directly affecting structural safety or traffic restriction decisions |
| Code basis and references | Cited AASHTO MBE/LRFD provisions, agency inspection manual sections, bridge plans, member IDs, photo numbers, calculation inputs, and line items in any worksheet or script must exactly match the current bridge file package |
| Severity rating | Severity claim must be consistent with the report's own observations and calculations, with no self-contradiction or exaggeration of condition state, load effect, remaining capacity, or urgency |
| Root cause analysis | Pinpoint the specific member, detail, deterioration mechanism, load path, boundary condition, or code check driving the concern and explain why it is structurally significant |
| Verification package | End-to-end calculation, field-evidence package, or inspection check that verifies the defect from the bridge owner's perspective, without altering photos, measurements, drawings, or rating assumptions just to manufacture the finding |
| Reproducibility | Verification can be followed end-to-end without manual assembly by the structural peer reviewer; structural-capacity or urgent-repair claims must be reproducible from the supplied measurements, plans, and calculations |
| Deduplication | Search prior inspection reports, maintenance work orders, load rating files, agency correspondence, and known repair plans to confirm the issue is not already documented, already mitigated, or already scheduled for official repair |
| Urgency amplification | The escalation must justify action beyond baseline routine monitoring, such as accelerated inspection, load posting, lane closure, shoring, emergency repair, or a materially different condition rating |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent bridge elements, plan sheets, inspection photos, code clauses, or AI-hallucinated calculation methods
- Severity rating contradicts the report's own observations or calculations
- Verification simulates the defect by altering source measurements, photos, drawings, or rating assumptions instead of using documented bridge conditions
- Code basis or bridge-file references inconsistent with the current bridge file package
- Verification package is incomplete and requires the structural peer reviewer to manually assemble missing calculations, measurements, or plan references
- Claimed urgency is equivalent to routine monitoring only, with no justified escalation beyond the existing inspection cycle

# f4: dispatch rules

Your only job is to alternate calls to the bridge inspection engineer and structural peer reviewer agents, driving the investigate → write → review → improve loop.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Call the bridge inspection engineer agent.
    2. Parse bridge inspection engineer's reply and extract the report file path.
    3. If a report path is present, call the structural peer reviewer agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the structural peer reviewer was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct concerns were investigated, and any anomalies encountered.

**Calling rules**

- Do **not** analyze bridge files, write reports, or do peer reviews yourself — delegate everything to the two sub-agents.
- If bridge inspection engineer or structural peer reviewer errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- bridge inspection engineer and structural peer reviewer each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`ideas.xml`, `reports/`, `counter.txt`).
- You don't need to understand any structural defects — just pass file paths correctly.
