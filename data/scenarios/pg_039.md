# background

I need you to set up a 3-agent pipeline for astronomical survey validation: survey coordinator, detection analyst, and false-positive referee.

- **survey coordinator** orchestrates: calls detection analyst, extracts the dossier file path from detection analyst's reply, then calls false-positive referee on that path, decrements a counter, and loops. Does no scientific analysis itself.
- **detection analyst** explores survey products, selects one exoplanet-candidate signal to investigate, and writes a candidate dossier file.
- **false-positive referee** evaluates the dossier against survey validation criteria and persists the verdict.

# f1: survey validation definition

You are participating in a wide-field astronomical survey validation campaign for exoplanet candidates.

Read the survey validation protocol first: ./survey-validation-protocol.md

# f2: dossier preparation tactics

This survey has already passed through mature automated vetting, human triage, and several catalog-level sanity checks. Any obvious false alarm would likely have been filtered out already. So:

- Don't prepare dossiers for problems that are obvious at a glance, such as a single bad cadence, a clearly saturated star, or a target already labeled as an eclipsing binary in the quick-look products
- Valuable candidate work comes from non-obvious signals: shallow transits near the noise floor, blended sources, grazing configurations, centroid shifts, ephemeris aliases, and subtle systematics that require careful cross-checking

**Writing the dossier**

Each candidate corresponds to its own dossier file at `dossiers/{id}-{slug}.md`, where `{id}` and `{slug}` come from the candidate's id and slug attributes in candidates.xml.

The dossier must contain four h2 sections in this fixed order:

- `## Dossier Title` — concise, clear candidate title including target identifier and likely planet signal
- `## Scientific Interest` — why the signal matters, such as possible small planet, temperate orbit, multi-planet architecture, or high-value follow-up target
- `## Signal Description` — short writeup of the observed transit-like feature, with references to light-curve files, periodogram outputs, aperture products, centroid diagnostics, and relevant catalog fields
- `## Reproduction Details` — runnable commands or notebook steps the referee can execute as-is to reproduce the folded light curve, transit fit, and diagnostic plots

Do not modify or delete the `## Review` section — the false-positive referee overwrites that themselves.

If after investigation you find the candidate doesn't hold, still briefly record your analysis and the reason for rejection in the dossier file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Candidate must come from the current survey release, or from an approved auxiliary light-curve product directly tied to this validation campaign |
| Data references | Cited target identifiers, sector/campaign numbers, cadence ranges, aperture files, catalog fields, and diagnostic products must exactly match the latest survey data release |
| Scientific interest | Claimed value must be consistent with the dossier's own analysis, with no self-contradiction or exaggerated planet interpretation |
| Signal analysis | Pinpoint the specific transit-like events, ephemeris, depth, duration, and model assumptions, and explain why the signal is plausibly astrophysical |
| Reproduction package | End-to-end script or notebook regenerating the folded light curve, transit model, odd/even comparison, centroid checks, and relevant plots from the referee's perspective |
| Reproducibility | Reproduction runs end-to-end without manual assembly by the referee; claimed periodic signals must recover the same period and epoch within the stated uncertainties |
| Deduplication | Search the survey vetting database, variable-star catalogs, eclipsing-binary catalogs, previous candidate lists, and public follow-up notes to confirm the signal is not already dispositioned or explained |
| False-positive discrimination | The dossier must show evidence stronger than “a dip exists”; it must address eclipsing binaries, background blends, instrumental artifacts, stellar variability, and noise/systematics |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent targets, files, sectors, catalog fields, or AI-hallucinated diagnostic products
- Scientific-interest claim contradicts the dossier's own analysis
- Reproduction simulates the transit by injecting or editing the observed light curve instead of analyzing survey data
- Data references inconsistent with the latest survey release
- Reproduction package is incomplete and requires the referee to manually assemble inputs
- Signal evidence is equivalent to random low-SNR noise or a known systematic, with no additional astrophysical support

# f4: dispatch rules

Your only job is to alternate calls to the detection analyst and false-positive referee agents, driving the explore → write → review → improve loop.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Call the detection analyst agent.
    2. Parse detection analyst's reply and extract the dossier file path.
    3. If a dossier path is present, call the false-positive referee agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the false-positive referee was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct candidates were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze light curves, write dossiers, or do reviews yourself — delegate everything to the two sub-agents.
- If detection analyst or false-positive referee errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- detection analyst and false-positive referee each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`candidates.xml`, `dossiers/`, `counter.txt`).
- You don't need to understand any astrophysical signals — just pass file paths correctly.
