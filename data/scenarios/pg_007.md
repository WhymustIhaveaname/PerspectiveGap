# background

I need you to set up a 5-agent pipeline that solves mathematical optimization problems by developing theoretical analysis and solver code in parallel, with each side informing the other. A problem arrives as a natural-language statement in `problem.md`. The pipeline alternates between revising the theory (bounds, invariants, structural claims) and revising the code (an implementation that achieves those guarantees).

The five agents:

- **dispatcher** orchestrates the run by alternating one theory step with one code step and tracking progress. It does no modeling, coding, or reviewing itself.
- **theory loop** — the **theorist** writes the theoretical analysis to `THEORY.md`; the **theory-reviewer** judges it. The theorist reads `SOLUTION.md` on each revision pass to incorporate empirical evidence.
- **code loop** — the **coder** writes solver code and its results to `SOLUTION.md`; the **code-reviewer** judges them. The coder reads `THEORY.md` on each revision pass to align the implementation with the latest theoretical claims.

At the end of every turn, each of the subagents briefly reports back what they did, what difficulties they hit, and any open questions.

The pipeline hands work off through three files:

- `problem.md` — the original problem, read-only. All four domain agents read it.
- `THEORY.md` — the theoretical analysis. Theorist writes; theory-reviewer reviews; coder and code-reviewer read it to know which bounds and invariants the implementation must respect.
- `SOLUTION.md` — the solver code and its results. Coder writes; code-reviewer reviews; theorist and theory-reviewer read it to ground the theory in observed numbers.

Each reviewer records its score as a `<review score=X>` block inside the file it reviewed, overwriting any previous block. The dispatcher reads those scores to drive the loops.

Unlike a plan-then-code pipeline where the plan must converge before the code starts, here neither side ever finishes first. Both loops alternate forever, capped at a fixed iteration count. Theory may tighten after seeing code's empirical numbers; code may correct after seeing theory's refined bounds.

# f1: problem context

You are working on a mathematical optimization problem, which may be linear, mixed-integer, nonlinear, or a combination. The original problem is given as a natural-language statement in `problem.md`; read it before starting your part.

# f2: theorist's deliverable

Produce the theoretical analysis for the problem and write it to `THEORY.md`. A complete analysis covers four kinds of claim, each justified with a proof sketch or a citation.

- **Bounds** — lower bounds on the objective from dual or LP relaxations or counting arguments; upper bounds from feasible constructions or known approximation ratios.
- **Structural properties** — convexity, monotonicity, submodularity, symmetries, decomposability into independent subproblems. State what holds, where, and under which assumptions.
- **Invariants** — quantities every feasible solution must preserve (conservation laws, parity constraints, total-mass conditions). The solver's output should satisfy them; the code-reviewer will check with `verify.py`.
- **Complexity or convergence** — worst-case complexity (P, NP-hard, FPT in some parameter), expected complexity for randomized methods, or convergence rate for iterative algorithms.

Ground your claims against `SOLUTION.md` if it has reported numbers. A predicted bound that is grossly off from observed numbers means either the theory needs revision or the code has a bug — diagnose which side has the issue and state it. Be concrete. "The objective is bounded below by the LP relaxation, value 73" is fine; "the problem has some structure that may help" is not.

# f3: theoretical analysis toolbox

Match the tool to the problem structure; do not force an ill-suited one. The list below is useful in theoretical analysis.

| Tool | Suitable for |
|---|---|
| Lagrangian relaxation / dual bounds | MILP, MINLP; lower bound on minimization, primal-dual gap analysis |
| KKT conditions | Differentiable NLP; certify local optimality of an interior point |
| Convex analysis (strong duality, convexity certificates) | LP, QP, convex NLP; global optimality from local |
| Monotonicity / submodularity | Combinatorial selection; (1-1/e) and similar greedy approximation ratios |
| Fixed-point theorems (Banach, Brouwer, Kakutani) | Iterative methods; existence of equilibrium, geometric convergence |
| Concentration inequalities (Hoeffding, Chernoff, Azuma) | Randomized algorithms; high-probability bounds on cost or error |
| Complexity reductions (NP-hardness, APX-hardness) | Exact-solution lower bounds; justify resorting to a heuristic |
| Parametric / sensitivity analysis | Stability of optimal value under perturbation of data |

# f4: writing the code

Read `problem.md` and `THEORY.md`, then implement the solver:

- Include every import. Create each variable with the correct type and bounds, define the objective and every constraint exactly as the problem states. Solve, check the solution status, and print both the solution and the objective value.
- Make the implementation testable against `THEORY.md`. If the theory states a dual bound B, your solver should achieve an objective at least B (for minimization). If the theory states an invariant, your output should satisfy it.
- Run the code yourself and watch it through to completion — do not stop while it is still running. If it errors, find and fix the root cause; do not bury errors under try/except.
- Sanity-check the answer against `THEORY.md`. A quantity that must be a whole number should not come back fractional; a bound the theory says holds should hold.

Write the code and its results up in `SOLUTION.md`.

# f5: solution doc discipline

Keep `SOLUTION.md` lean. Its body — everything except the `<review>` block — must stay under 3000 words (~20KB), about a four-page note; record only the final state and move iteration-by-iteration history into a separate `CHANGELOG.md`. Bold text must not exceed 5% of the file: if everything is bold, nothing is.

# f6: theory review

Review `THEORY.md` against `problem.md` and against the numbers reported in `SOLUTION.md` (if any).

- **Faithfulness** — does the theory correctly describe the problem stated in `problem.md`? Check definitions, decision variable types, constraint translations.
- **Proof soundness** — for each claim, walk through the proof sketch or citation. Flag hand-waving steps. Verify cited inequalities apply under the stated assumptions.
- **Empirical grounding** — open `SOLUTION.md`. If the code reports objective values, do they fall within the theory's predicted bounds? A dual bound that is loose by orders of magnitude is a signal the relaxation is too weak; tighten it, or note explicitly that the bound is only illustrative.

Be brief and specific. Do not write theory yourself — only review. If a claim is wrong, say which one and why.

# f7: code review

Review `SOLUTION.md` against `problem.md` and against `THEORY.md`.

- Inspect the reported result for anomalies, unreasonable objective values, or constraint violations stated directly in `problem.md`.
- Open `THEORY.md`. For every bound and invariant the theory claims, write a check into a separate `verify.py`: does the solver's output satisfy the invariant, fall within the predicted bound, achieve the predicted approximation ratio? Run it. Name any variable or quantity that violates.
- Read the solver code and spot-check it for bugs — unused parameters, dead code, logic that contradicts the chosen algorithm. A bug that silently degrades the solution is worse than one that crashes.
- Treat the result as finished only when every constraint holds, every theory-claimed invariant holds, and the objective is consistent with the theory's bounds — not merely when the code runs without error.
- Check the length of `SOLUTION.md`: if it exceeds 20KB, give a warning, but do not deduct points for it.
- Check the bold ratio of `SOLUTION.md`: if bold text exceeds 5% of the file, give a warning, but do not deduct points for it.

# f8: scoring

Score the work from 1 to 10, and record your verdict by overwriting the single `<review score=X>` block at the end of the document you reviewed — overwrite the existing one, never append a second.

Score strictly on real merit. Do not give any credit for a piece of work merely being honest about its own shortcomings — "this doesn't work, but I'm telling you it doesn't" is still a failing result, not a passing one.

# f9: orchestration

You orchestrate the run; you never model, code, or review yourself — you only dispatch the right agent at the right time.

- **Init** — create `STATE.md` with `iter: 0` and `last_side: none`. Dispatch the theorist for the first theory step (theorist, then theory-reviewer), then the coder for the first code step (coder, then code-reviewer).
- **Alternation loop** — alternate one theory step with one code step. Neither side terminates early on its own score; even a perfect-looking theory may need to revise once the code reports actual numbers.
- Cap the total at 20 iterations (10 theory steps + 10 code steps). After the cap, stop and summarize.
- Keep the current iteration count, last side, and the two latest scores in `STATE.md`, accurate after each dispatch. Read each score from the reviewer's `<review score=X>` block. Commit progress after each step.
- Read each agent's brief narrative reply on return. If an agent reports an unresolved error, append it to `error-log.txt` (timestamp + agent name + content) and continue with the next step.

# f10: theory revision pass

The theoretical analysis lives in `THEORY.md`, and you may be invoked on it more than once — the pipeline runs you and the theory-reviewer as a loop, alternating with the code loop.

- If `THEORY.md` does not exist yet, this is the first pass: produce the analysis from scratch based on `problem.md`.
- If `THEORY.md` already exists, it carries the theory-reviewer's `<review>` block. Read that feedback. Then open `SOLUTION.md` and look at the latest reported numbers — they may invalidate, sharpen, or motivate revisions to your claims. Revise `THEORY.md` in place to address every reviewer point raised and to incorporate any empirical evidence that changes your analysis. Don't discard the previous analysis and start over, and don't leave a flagged issue unaddressed.

# f11: code revision pass

The solver code and its results live in `SOLUTION.md`, and you may be invoked on it more than once — the pipeline runs you and the code-reviewer as a loop, alternating with the theory loop.

- If `SOLUTION.md` does not exist yet, this is the first pass: produce the solution from scratch based on `problem.md`.
- If `SOLUTION.md` already exists, it carries the code-reviewer's `<review>` block. Read that feedback. Then open `THEORY.md` and check whether the latest theory introduces bounds, invariants, or structural claims your current implementation doesn't yet respect — for example a tighter dual bound your solver is now far from achieving, or an invariant your output violates. Revise `SOLUTION.md` in place to address every reviewer point and to align with the latest theory. Don't discard the previous solution and start over, and don't leave a flagged issue unaddressed.

# f12: report back

When you finish, briefly report to the dispatcher: what you did, what difficulties you hit (or didn't resolve), and any open questions.
