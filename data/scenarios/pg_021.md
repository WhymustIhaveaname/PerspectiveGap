# background

I need you to set up a 2-agent pipeline for pharmaceutical clinical trial data submission. The two roles are clinical_researcher and regulatory_reviewer: clinical_researcher is responsible for analyzing trial data to establish drug efficacy and identify safety signals, while regulatory_reviewer is responsible for strictly auditing the submission to uncover statistical p-hacking, hidden safety signals, and flawed methodology. The flow is roughly:

```
WHILE TRUE:
    1. clinical_researcher picks a hypothesis it likes and explores it deeply, or appends more hypotheses to hypotheses.xml and then picks one to explore.
    2. clinical_researcher spawns a **fresh** regulatory_reviewer to evaluate it.
```

# f1: problem definition

You are participating in a pharmaceutical clinical trial data submission and strict regulatory review process.

Read the regulatory submission guidelines first: ./regulatory-guidelines.md

# f2: analysis tactics

This clinical trial has been analyzed extensively for years — any obvious major efficacy signal without safety trade-offs would have been proven long ago. So:

- Don't report findings that are superficial or ignore key covariates (e.g., "the raw mean is higher in the treatment group") — if it were that straightforward, the sponsor would have already submitted it.
- Truly valuable insights and critical safety signals hide in complex subgroup interactions, longitudinal data trends, and boundary conditions of the statistical models.

**Writing the submission report**

Each hypothesis corresponds to its own report file at `reports/{id}-{slug}.md`, where `{id}` and `{slug}` come from the hypothesis's id and slug attributes in hypotheses.xml.

The report must contain four h2 sections in this fixed order:

- `## Report Title` — concise, clear summary of the clinical finding
- `## Clinical Impact Description` — what this means for patient outcomes (efficacy benefits, adverse event risks, therapeutic window, etc.)
- `## Statistical Description` — short writeup of the methodology used, with dataset/variable references
- `## Reproduction Details` — a runnable SAS/R script the regulatory_reviewer can execute as-is to reproduce the statistical analysis

Do not modify or delete the `## Regulatory Review` section — the regulatory_reviewer overwrites that themselves.

If after investigation you find the hypothesis doesn't yield statistically significant or clinically meaningful results, still briefly record your analysis and the reason for rejection in the report file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within the pre-specified primary or secondary endpoints, or explicitly declared as an exploratory post-hoc analysis |
| Data references | Cited datasets, variable names, and patient counts must exactly match the locked trial database |
| Impact description | Clinical impact claim consistent with the report's statistical results, no self-contradiction, no exaggeration |
| Statistical rigor | Pinpoint specific models used, verify assumptions (e.g., proportional hazards, normal distribution), and check for signs of p-hacking |
| Reproducibility script | End-to-end analysis script generating the exact p-values and confidence intervals from the raw data (must not manually alter or impute missing data without declaring) |
| Execution | Script runs end-to-end without manual adjustments by the regulatory_reviewer; standard models must run within < 10 minutes on the provided compute environment |
| Deduplication | Search previous submission documents, interim analyses, and protocol amendments to confirm this isn't a recycled or already rejected hypothesis |
| Clinical significance | The effect size must exceed the minimal clinically important difference, not just achieving bare statistical significance |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent datasets or AI-hallucinated variable names
- Impact description contradicts the report's own statistical results
- Reproducibility script achieves significance by silently dropping unfavorable patient cohorts (p-hacking)
- Data references inconsistent with the locked trial database
- Analysis script is incomplete and requires the regulatory_reviewer to manually assemble or debug it
- The effect is statistically significant but clinically meaningless, with no actual patient benefit

# f4: dispatch rules

After compiling a submission report, submit it to a **fresh** regulatory_reviewer for review. Do not review your own report.

**After the review returns:** execute all action steps in the regulatory_reviewer's output. Then immediately go back to step 1: re-read hypotheses.xml and pick the next hypothesis by UCB to keep working. Don't wait for user instructions, don't stop. One round = write report → review → execute actions → back to step 1.

2. Update the hypothesis's status in hypotheses.xml: increment attempts by 1, then update score.

# f6: hypothesis selection

Before starting work, read hypotheses.xml.

1. Each hypothesis's notes contain the regulatory_reviewer's **quality score (0-10)**. A score of 0 means the hypothesis is dead — don't touch it again. `none` means it hasn't been attempted yet.
2. Prefer hypotheses with `attempts="0"` and `score="none"` first.
3. Otherwise sort by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; higher score with fewer attempts ranks first.
4. If no hypothesis is worth trying, search the clinical trial database and literature for new hypotheses, append them to hypotheses.xml, then pick one to start.

**Recording rule during exploration**

While exploring trial data, the moment you spot any potential new efficacy or safety signal — even if you're unsure it holds — immediately append it to hypotheses.xml (`score="none"`, `attempts=0`). Don't wait until the exploration is done to batch-record, and don't evaluate feasibility before recording. Recording ≠ committing to attempt it, just preventing forgetting. Evaluation belongs to the selection phase.
