# background

I need you to set up a 3-agent pipeline for patent validity challenge preparation: matter coordinator, patent search specialist, and invalidity reviewer.

- **matter coordinator** orchestrates: each round it selects which prior-art theory to pursue next, calls the patent search specialist on that theory, extracts the claim-chart file path from the patent search specialist's reply, then calls the invalidity reviewer on that path, decrements a counter, and loops. Does no patent analysis itself.
- **patent search specialist** investigates the patent, asserted claims, prosecution history, and prior-art sources for the theory it is given, writes an invalidity chart file, and records any new prior-art theories or claim attack surfaces it notices along the way.
- **invalidity reviewer** evaluates the chart against validity-challenge criteria and persists the verdict.

# f1: challenge definition

You are preparing materials for a patent validity challenge against asserted claims in a technology patent.

Read the challenge rules and claim-construction assumptions first: ./challenge-rules.md

# f2: search and charting tactics

Assume the patent has already gone through prosecution, portfolio diligence, and at least a basic prior-art search. Any facially obvious invalidity argument may already have been considered. So:

- Don't chart shallow arguments that are apparent from the patent's front page or a keyword-only search — if it were that obvious, opposing counsel or the examiner likely already addressed it
- Valuable invalidity theories usually hide in non-obvious references, overlooked priority problems, inherency, standards documents, product manuals, archived web materials, or combinations whose motivation to combine is well supported

**Writing the chart**

Each theory corresponds to its own chart file at `charts/{id}-{slug}.md`, where `{id}` and `{slug}` come from the theory's id and slug attributes in theories.xml.

The chart must contain four h2 sections in this fixed order:

- `## Chart Title` — concise, clear invalidity theory title
- `## Invalidity Rationale` — what basis it supports, such as anticipation, obviousness, lack of written description, or priority-date defeat
- `## Short Description` — short writeup of the theory, with claim numbers, limitation identifiers, and prior-art references
- `## Evidence and Mapping Details` — quotations, page/line/figure citations, archive links, and a limitation-by-limitation mapping sufficient for the invalidity reviewer to verify

Do not modify or delete the `## Review` section — the invalidity reviewer overwrites that themselves.

If after investigation you find the theory doesn't hold, still briefly record your analysis and the reason for rejection in the chart file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Addresses the asserted patent, challenged claims, and permitted invalidity grounds for this matter |
| Reference accuracy | Prior-art citations, page numbers, figure numbers, publication dates, and quoted passages must exactly match the cited source copies |
| Invalidity rationale | Legal theory is consistent with the chart's own evidence, no self-contradiction, no overclaiming |
| Element mapping | Pinpoint each claim limitation and explain where it is disclosed or why it would have been obvious |
| Evidence package | Includes source copies, archive records where needed, translations if applicable, and enough context to verify the mapping without guesswork |
| Verifiability | Materials can be checked end-to-end without manual reconstruction by the invalidity reviewer; priority, public availability, and claim-version assumptions are stated |
| Deduplication | Search prior IPRs, litigation filings, prosecution history, examiner-cited art, and known invalidity contentions to confirm the argument is not already known or already disposed of |
| Argument strength | The theory must materially improve the validity challenge beyond merely naming a cumulative reference or repeating an examiner-considered argument |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent prior art, fabricated publication dates, or AI-hallucinated quotations
- Invalidity rationale contradicts the chart's own limitation mapping
- Chart relies on a reference that post-dates the applicable priority date without explaining why that is permissible
- Citations inconsistent with the source copies or asserted claim version
- Evidence package is incomplete and requires the invalidity reviewer to assemble missing materials
- Argument is merely cumulative of examiner-considered art or previously served invalidity contentions, with no material improvement

# f4: coordination rules

You drive the select → search → review → improve loop: each round, select the next prior-art theory, then alternate calls to the patent search specialist and invalidity reviewer agents.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Select the next theory, then call the patent search specialist agent with it.
    2. Parse patent search specialist's reply and extract the chart file path.
    3. If a chart path is present, call the invalidity reviewer agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the invalidity reviewer was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct theories were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze claims, search prior art, write charts, or do reviews yourself — delegate everything to the two sub-agents.
- If patent search specialist or invalidity reviewer errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- patent search specialist and invalidity reviewer each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`theories.xml`, `charts/`, `counter.txt`).
- You don't need to understand any invalidity theories — just pass file paths correctly.

# f5: where to save review results

After completing the review, you must do two things:

1. Update the chart file's Review section

At the end of the chart file, write or overwrite the `## Review` section (always keep exactly one; replace it if it already exists). Follow this structure:

```
## Review

**Score: X/10** (one-sentence reason)

### Fatal flaws (if any)
- 🔴 ...

### Per-dimension review

| Dimension | Verdict | Notes |
|-----------|---------|-------|
| Scope | pass/fail | ... |
| Reference accuracy | pass/fail | ... |
| Invalidity rationale | pass/fail | ... |
| Element mapping | pass/fail | ... |
| Evidence package | pass/fail | ... |
| Verifiability | pass/fail | ... |
| Deduplication | pass/fail | ... |
| Argument strength | pass/fail | ... |

### Improvement suggestions
Be precise about which part of the chart needs changes. If the theory doesn't hold at all, explicitly recommend abandoning it.
```

2. Update the theory's status in theories.xml: increment attempts by 1, then update score.

# f6: report back

When you finish, briefly report to the matter coordinator what you did.

# f7: theory selection

Each round, before calling the patent search specialist, read theories.xml and select which theory the patent search specialist should work on:

1. Every theory carries an `attempts` count and an invalidity reviewer `score` (0-10). `score="0"` means the theory is dead — never select it again. `score="none"` means it has not been attempted yet.
2. Prefer theories with `attempts="0"` and `score="none"` first.
3. Otherwise rank by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; a higher score with fewer attempts ranks first.
4. Pass the selected theory to the patent search specialist. If theories.xml holds no workable theory, call the patent search specialist with none and let it explore and surface fresh ones.

# f8: recording new theories

While exploring the patent, prosecution history, asserted claim set, prior-art databases, standards archives, product documentation, or litigation materials, the moment you spot any potential new invalidity theory or claim attack surface — even if you are unsure it holds — immediately append it to theories.xml (`score="none"`, `attempts="0"`). Don't wait until exploration is finished to batch-record, and don't judge feasibility before recording. Recording a theory is not a commitment to pursue it; it just keeps the lead from being forgotten. Whether it ever gets worked on is decided later, when the next theory is selected.
