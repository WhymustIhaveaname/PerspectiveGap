# background

I need you to set up a 5-agent pipeline that solves mathematical optimization problems. A problem arrives as a natural-language statement in `problem.md`, and the pipeline turns it into a vetted plan and then working, verified solver code.

The five agents:

- **dispatcher** orchestrates the run: it drives the plan loop, then the code loop, and tracks progress. It does no modeling, coding, or reviewing itself.
- **plan loop** — the **plan-creator** turns the problem into a formal model and a solving strategy; the **plan-critic** reviews it. They iterate until the plan is sound.
- **code loop** — the **coder** implements the strategy as solver code; the **code-critic** reviews the code and its result. They iterate until the result is correct.

The pipeline hands work off through three files:

- `problem.md` — the original problem, given as input. All four domain agents (plan-creator, plan-critic, coder, code-critic) read it.
- `PLAN.md` — the plan. The plan-creator writes it; the plan-critic reviews it; the coder reads it to implement it.
- `SOLUTION.md` — the solver code and its results. The coder writes it; the code-critic reviews it.

Each critic records its score as a `<review score=X>` block inside the file it reviewed, and the dispatcher reads those scores to drive the loops.

# f1: problem context

You are working on a mathematical optimization problem — it may be linear, mixed-integer, nonlinear, or a combination. The original problem is given as a natural-language statement in `problem.md`; read it before starting your part.

# f2: plan-creator's deliverable

Produce the plan for the problem and write it to `PLAN.md`. A complete plan has two parts — a formalized model and one chosen solving strategy.

The **model** must state:

- **Problem type** — LP, MILP, NLP, MINLP, QP, and so on; this must be exact.
- **Data** — the key numeric inputs and what each one represents.
- **Decision variables** — their types and domains, with descriptive names.
- **Objective function** — the exact expression to optimize.
- **Constraints** — every constraint, listed explicitly.

The **strategy** — weigh the options and commit to the single most promising route — must state:

- which solver to use and how to call it;
- the algorithm and its key initial settings;
- the reformulation or trick the route relies on;
- the biggest risk on this route.

# f3: solver toolbox

Which solver suits which kind of problem — use it to choose one for the strategy:

- **PuLP** — LP and MILP only; no nonlinear support.
- **Pyomo** — LP, MILP, NLP, MINLP; the broadest coverage, and solver-independent.
- **OR-Tools** — LP, MILP, constraint programming, and routing.
- **Gurobi** — top performance on MILP and convex/nonconvex QP; commercial license; not for general NLP.
- **SCIP** — the strongest open-source option for MINLP and global optimization; on pure MILP, slower than Gurobi.
- **IPOPT** — large-scale nonlinear problems.

Match the solver to the problem type; do not force an ill-suited one.

# f4: writing the code

Read the plan in `PLAN.md` and implement its strategy as solver code:

- Include every import. Create each variable with the correct type and bounds, define the objective and every constraint exactly as the model states, solve, check the solution status, and print both the solution and the objective value.
- Run the code yourself and watch it through to completion — do not stop while it is still running. If it errors, find and fix the root cause; do not bury errors under try/except.
- Sanity-check the answer: a quantity that must be a whole number should not come back fractional.

Write the code and its results up in `SOLUTION.md`.

# f5: solution doc discipline

Keep `SOLUTION.md` lean. Its body — everything except the `<review>` block — must stay under 3000 words (~20KB), about a four-page note; record only the final state and move iteration-by-iteration history into a separate `CHANGELOG.md`. Bold text must not exceed 5% of the file: if everything is bold, nothing is.

# f6: plan review

Review the plan in `PLAN.md`:

- **Model** — does it faithfully capture the problem? Check that the problem type, decision variables, objective, and constraints are all accurate.
- **Strategy** — is the chosen solver and algorithm a sound fit, is the route actually implementable, and is it likely to reach a correct solution?

Be brief and specific. Do not write any solver code yourself — only review.

# f7: code review

Review `SOLUTION.md` — the solver code and its results. Verify it directly against the problem statement.

- Inspect the reported result for anomalies, unreasonable objective values, or constraint violations.
- Write a separate `verify.py` that re-checks every constraint against the reported solution, and run it; name any variables that violate a constraint.
- Read the solver code and spot-check it for bugs — unused parameters, dead code, logic that contradicts what the solution says it does. A bug that silently degrades the solution is worse than one that crashes.
- Treat the result as finished only when every constraint holds and the objective value is sound — not merely when the code runs without error.
- Check the length of `SOLUTION.md`: if it exceeds 20KB, give a warning, but do not deduct points for it.
- Check the bold ratio of `SOLUTION.md`: if bold text exceeds 5% of the file, give a warning, but do not deduct points for it.

# f8: scoring

Score the work from 1 to 10, and record your verdict by overwriting the single `<review score=X>` block at the end of the document you reviewed — overwrite the existing one, never append a second.

Score strictly on real merit. Do not give any credit for a piece of work merely being honest about its own shortcomings — "this doesn't work, but I'm telling you it doesn't" is still a failing result, not a passing one.

# f9: orchestration

You orchestrate the run; you never model, code, or review yourself — you only dispatch the right agent at the right time.

- **Init** — dispatch the plan-creator to draft the plan.
- **Plan loop** — have the plan-creator and plan-critic iterate on the model and strategy until the plan-critic scores the plan 10.
- **Code loop** — once the plan is settled, have the coder and code-critic iterate on the implementation until the code-critic scores it 10.
- Cap each loop at 10 iterations.
- Keep the current phase, iteration count, and latest score in `STATE.md`, accurate after each dispatch. Read each score from the critic's `<review score=X>` block. Commit progress after each step.

# f10: plan revision pass

The plan lives in `PLAN.md`, and you may be invoked on it more than once — the pipeline runs you and the plan-critic as a loop.

- If `PLAN.md` does not exist yet, this is the first pass: produce the plan from scratch.
- If `PLAN.md` already exists, it carries the plan-critic's `<review>` block: read that feedback and revise `PLAN.md` in place to address every point raised. Don't discard the plan and start over, and don't leave a flagged issue unaddressed.

# f11: code revision pass

The solver code and its results live in `SOLUTION.md`, and you may be invoked on it more than once — the pipeline runs you and the code-critic as a loop.

- If `SOLUTION.md` does not exist yet, this is the first pass: produce the solution from scratch.
- If `SOLUTION.md` already exists, it carries the code-critic's `<review>` block: read that feedback and revise `SOLUTION.md` in place to address every point raised. Don't discard the solution and start over, and don't leave a flagged issue unaddressed.
