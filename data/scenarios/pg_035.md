# background

I need you to set up a 3-agent pipeline for pharmacovigilance signal assessment: case coordinator, safety signal analyst, and medical safety reviewer.

- **case coordinator** orchestrates: calls safety signal analyst, extracts the dossier file path from safety signal analyst's reply, then calls medical safety reviewer on that path, decrements a counter, and loops. Does no analysis itself.
- **safety signal analyst** explores the safety data package, picks a candidate adverse-drug-reaction signal to investigate, writes a signal dossier file.
- **medical safety reviewer** evaluates the dossier against pharmacovigilance criteria and persists the verdict.

# f1: problem definition

You are participating in a pharmacovigilance signal assessment workflow for marketed medicines.

Read the signal assessment charter first: ./signal-assessment-charter.md

# f2: signal investigation and dossier drafting tactics

This product portfolio has been monitored for many years — any obvious labeled adverse reaction or well-known class effect should already be documented. So:

- Don't escalate observations that are obvious at a glance (e.g. "nausea is reported with the drug") — if it were that obvious, it would likely already be in labeling, aggregate reports, or routine safety surveillance
- Truly valuable safety signals hide in non-obvious patterns: unusual time-to-onset, plausible dechallenge/rechallenge clusters, high-quality serious cases, vulnerable subpopulations, medication-error pathways, or shifts across data sources

**Writing the dossier**

Each candidate corresponds to its own dossier file at `dossiers/{id}-{slug}.md`, where `{id}` and `{slug}` come from the candidate's id and slug attributes in signal-candidates.xml.

The dossier must contain four h2 sections in this fixed order:

- `## Signal Title` — concise, clear adverse-drug-reaction signal title
- `## Public Health Impact Description` — what harm it can cause, including seriousness, reversibility, preventability, and potential exposed population
- `## Short Description` — short writeup of the suspected association, with references to source tables, case narratives, MedDRA terms, and relevant product history
- `## Evidence Details` — traceable evidence the medical safety reviewer can inspect as-is, including case identifiers, query definitions, observed/expected counts if applicable, and narrative summaries

Do not modify or delete the `## Review` section — the medical safety reviewer overwrites that themselves.

If after investigation you find the candidate signal doesn't hold, still briefly record your analysis and the reason for rejection in the dossier file.

# f3: medical safety review criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within post-marketing pharmacovigilance for the assigned product or product class, or outside it but directly affecting the product's safety profile |
| Source references | Cited case IDs, MedDRA preferred terms, data-lock point, query definitions, and table numbers must exactly match the latest safety data package |
| Public health impact description | Impact claim consistent with the dossier's own analysis, no self-contradiction, no exaggeration beyond seriousness, frequency, preventability, or exposed-population evidence |
| Causality analysis | Pinpoint the specific clinical and temporal evidence supporting or weakening causality, including alternative etiologies, confounding by indication, concomitant medicines, dechallenge, and rechallenge where available |
| Evidence package | End-to-end traceable case series or quantitative signal output from the reporter's perspective; must not rely on invented cases, untraceable summaries, or post hoc recoding to create the signal |
| Reproducibility | Evidence can be followed end-to-end without manual reconstruction by the medical safety reviewer; database queries must specify data source, data-lock date, product terms, event terms, seriousness filters, and deduplication logic |
| Deduplication | Search labeling, periodic safety reports, risk management plans, health-authority requests, literature, and prior internal signal logs to confirm the signal is not already characterized or closed with adequate rationale |
| Regulatory significance | The proposed signal must exceed the baseline of isolated expected background reports and plausibly warrant additional pharmacovigilance action, labeling consideration, targeted follow-up, or risk minimization discussion |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent cases, invented database fields, or AI-hallucinated MedDRA terms
- Public health impact description contradicts the dossier's own analysis
- Evidence package creates the signal by recoding, excluding, or editing source cases without justification
- Source references inconsistent with the latest safety data package
- Evidence is incomplete and requires the medical safety reviewer to manually assemble the case series or query
- Proposed effect is equivalent to isolated expected background reporting, with no added regulatory or clinical significance

# f4: case coordination rules

Your only job is to alternate calls to the safety signal analyst and medical safety reviewer agents, driving the investigate → write → review → improve loop.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Call the safety signal analyst agent.
    2. Parse safety signal analyst's reply and extract the dossier file path.
    3. If a dossier path is present, call the medical safety reviewer agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the medical safety reviewer was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct candidate signals were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze safety data, write dossiers, or do reviews yourself — delegate everything to the two sub-agents.
- If safety signal analyst or medical safety reviewer errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- safety signal analyst and medical safety reviewer each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`signal-candidates.xml`, `dossiers/`, `counter.txt`).
- You don't need to understand any adverse-drug-reaction signals — just pass file paths correctly.

**Score: X/10** (one-sentence reason)

### Fatal flaws (if any)
