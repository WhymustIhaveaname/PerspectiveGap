# background

I need you to set up a 5-agent cybersecurity pipeline that assesses target systems by developing abstract threat models and concrete red-team exploitation scripts in parallel, with each side informing the other. A system assessment request arrives as a natural-language description in `problem.md`. The pipeline alternates between revising the threat model (attack vectors, invariants, structural vulnerabilities) and revising the exploit scripts (a concrete implementation that tests those defenses).

The five agents:

- **secops-director** orchestrates the run by alternating one threat model step with one exploit step and tracking progress. It does no modeling, scripting, or reviewing itself.
- **threat modeling loop** — the **threat-modeler** writes the abstract threat model to `THEORY.md`; the **security-architect** judges it. The threat-modeler reads `SOLUTION.md` on each revision pass to incorporate empirical execution results from the red team.
- **exploit loop** — the **red-teamer** writes exploit scripts and execution results to `SOLUTION.md`; the **vulnerability-analyst** judges them. The red-teamer reads `THEORY.md` on each revision pass to align the attack paths with the latest theoretical vulnerabilities.

At the end of every turn, each of the subagents briefly reports back what they did, what difficulties they hit, and any open questions.

The pipeline hands work off through three files:

- `problem.md` — the original target system description, read-only. All four domain agents read it.
- `THEORY.md` — the abstract threat model. Threat-modeler writes; security-architect reviews; red-teamer and vulnerability-analyst read it to know which security invariants and attack paths the scripts must evaluate.
- `SOLUTION.md` — the exploit scripts and their results. Red-teamer writes; vulnerability-analyst reviews; threat-modeler and security-architect read it to ground the theoretical model in observed execution data.

Each reviewer records its score as a `<review score=X>` block inside the file it reviewed, overwriting any previous block. The secops-director reads those scores to drive the loops.

Unlike a linear assessment pipeline where the model must converge before exploitation starts, here neither side ever finishes first. Both loops alternate forever, capped at a fixed iteration count. The threat model may tighten after seeing the exploit's empirical hurdles; the exploit script may pivot after seeing the threat model's refined attack surfaces.

# f1: target system context

You are working on a cybersecurity assessment for a target system, which may be web infrastructure, compiled binaries, network architecture, or a combination. The original system description is given as a natural-language statement in `problem.md`; read it before starting your part.

# f2: threat modeler's deliverable

Produce the abstract threat model for the target system and write it to `THEORY.md`. A complete model covers four kinds of claim, each justified with an architectural proof sketch or a vulnerability citation.

- **Attack Surface & Vectors** — exposed APIs, network ingress points, user trust boundaries, and specific entry parameters.
- **Security Properties** — privilege separation, encryption guarantees, sanitization structural claims, and trust models. State what defenses hold, where they exist, and under which assumptions.
- **Security Invariants** — conditions that the system must maintain (e.g., unauthorized users cannot read protected files, sessions must isolate state, memory boundaries hold). The exploit's output should test these limits; the vulnerability-analyst will check with `verify.py`.
- **Impact & Complexity** — worst-case impact (RCE, data exfiltration, DoS), prerequisite complexity, expected execution time for brute-force methods, or CVSS estimations.

Ground your claims against `SOLUTION.md` if it has reported numbers or execution traces. A predicted attack vector that completely fails to produce a viable exploit means either the threat model needs revision or the script has a bug — diagnose which side has the issue and state it. Be concrete. "The authentication mechanism is vulnerable to a timing attack, succeeding in ~200ms" is fine; "the system has some weaknesses that may help" is not.

# f3: threat modeling toolbox

Match the tool to the target system's architecture; do not force an ill-suited one. The list below is useful in threat modeling and vulnerability analysis.

| Tool | Suitable for |
|---|---|
| Data Flow Analysis | Web/Network architectures; tracing untrusted input to sensitive sinks |
| Attack Trees | Complex systems; modeling multi-step kill chains and prerequisite paths |
| STRIDE Threat Categorization | System architecture; systematic identification of spoofing, tampering, etc. |
| Privilege Escalation Matrices | IAM / RBAC; identifying lateral movement and vertical escalation routes |
| Cryptographic Protocol Analysis | Auth/Encryption; verifying key exchange and secrecy properties |
| Fuzzing Heuristic Modeling | Memory-unsafe code; predicting boundary conditions and crash patterns |
| Taint Analysis | AppSec/Binaries; tracking malicious payload propagation through memory |
| Exploit Mitigation Bypass Analysis | Defended environments; structuring payloads around ASLR/DEP/WAF |

# f4: writing the exploit scripts

Read `problem.md` and `THEORY.md`, then implement the red-team exploit scripts:

- Include every import. Create each payload with the correct encoding, define the network targets and memory offsets exactly as the threat model outlines. Execute the script, check the response status, and print both the execution trace and the extracted/compromised data.
- Make the implementation testable against `THEORY.md`. If the model states an attack path requires administrative access, your script should demonstrate acquiring it. If the model defines a security invariant, your script should attempt to bypass or violate it.
- Run the code yourself and watch it through to completion — do not stop while it is still running. If it errors, find and fix the root cause; do not bury errors under try/except.
- Sanity-check the result against `THEORY.md`. A vulnerability that is supposed to yield a root shell should not just return a low-privilege token; a memory leak the threat model says is accessible should actually be readable.

Write the code and its execution results up in `SOLUTION.md`.

# f5: exploit doc discipline

Keep `SOLUTION.md` lean. Its body — everything except the `<review>` block — must stay under 3000 words (~20KB), about a four-page note; record only the final exploitation state and move iteration-by-iteration history into a separate `CHANGELOG.md`. Bold text must not exceed 5% of the file: if everything is bold, nothing is.

# f6: threat model review

Review `THEORY.md` against `problem.md` and against the execution traces reported in `SOLUTION.md` (if any).

- **Faithfulness** — does the threat model correctly describe the architecture stated in `problem.md`? Check component definitions, trust boundaries, and network topologies.
- **Soundness** — for each attack path, walk through the logic. Flag hand-waving steps. Verify cited vulnerabilities apply under the stated system versions and configurations.
- **Empirical grounding** — open `SOLUTION.md`. If the script reports execution outcomes, do they align with the threat model's predicted impact and complexity? An estimated attack complexity that is orders of magnitude off from actual execution time is a signal the model's assumptions are flawed; correct them, or note explicitly that the theoretical complexity is only illustrative.

Be brief and specific. Do not write the threat model yourself — only review. If an attack vector is fundamentally unworkable, say which one and why.

# f7: exploit review

Review `SOLUTION.md` against `problem.md` and against `THEORY.md`.

- Inspect the reported execution result for anomalies, failed network connections, or missed targets stated directly in `problem.md`.
- Open `THEORY.md`. For every security invariant the threat model claims, write a check into a separate `verify.py`: does the red-team script's output demonstrably break the invariant, bypass the predicted control, or achieve the modeled impact? Run it. Name any mechanism or state that violates the threat model's expectations.
- Read the exploit code and spot-check it for bugs — unused payloads, dead logic, or mechanisms that contradict the chosen attack vector. A script bug that silently fails to deliver the payload is worse than one that crashes.
- Treat the result as finished only when every targeted access constraint is tested, every threat-model-claimed invariant is evaluated, and the impact is consistent with the model's predictions — not merely when the script runs without error.
- Check the length of `SOLUTION.md`: if it exceeds 20KB, give a warning, but do not deduct points for it.
- Check the bold ratio of `SOLUTION.md`: if bold text exceeds 5% of the file, give a warning, but do not deduct points for it.

# f8: scoring

Score the work from 1 to 10, and record your verdict by overwriting the single `<review score=X>` block at the end of the document you reviewed — overwrite the existing one, never append a second.

Score strictly on real merit. Do not give any credit for a piece of work merely being honest about its own shortcomings — "this exploit script fails, but I'm telling you it fails" is still a failing result, not a passing one.

# f9: orchestration

You orchestrate the run; you never model, script, or review yourself — you only dispatch the right agent at the right time.

- **Init** — create `STATE.md` with `iter: 0` and `last_side: none`. Dispatch the threat-modeler for the first threat model step (threat-modeler, then security-architect), then the red-teamer for the first exploit step (red-teamer, then vulnerability-analyst).
- **Alternation loop** — alternate one threat model step with one exploit step. Neither side terminates early on its own score; even a perfect-looking threat model may need to revise once the exploit script reports actual execution outcomes.
- Cap the total at 20 iterations (10 threat model steps + 10 exploit steps). After the cap, stop and summarize.
- Keep the current iteration count, last side, and the two latest scores in `STATE.md`, accurate after each dispatch. Read each score from the reviewer's `<review score=X>` block. Commit progress after each step.
- Read each agent's brief narrative reply on return. If an agent reports an unresolved error, append it to `error-log.txt` (timestamp + agent name + content) and continue with the next step.

# f10: threat model revision pass

The abstract threat model lives in `THEORY.md`, and you may be invoked on it more than once — the pipeline runs you and the security-architect as a loop, alternating with the exploit loop.

- If `THEORY.md` does not exist yet, this is the first pass: produce the threat model from scratch based on `problem.md`.
- If `THEORY.md` already exists, it carries the security-architect's `<review>` block. Read that feedback. Then open `SOLUTION.md` and look at the latest reported execution traces — they may invalidate, sharpen, or motivate revisions to your attack paths. Revise `THEORY.md` in place to address every reviewer point raised and to incorporate any empirical execution data that changes your model. Don't discard the previous model and start over, and don't leave a flagged issue unaddressed.

# f11: exploit revision pass

The exploit scripts and their results live in `SOLUTION.md`, and you may be invoked on it more than once — the pipeline runs you and the vulnerability-analyst as a loop, alternating with the threat modeling loop.

- If `SOLUTION.md` does not exist yet, this is the first pass: produce the exploit script from scratch based on `problem.md`.
- If `SOLUTION.md` already exists, it carries the vulnerability-analyst's `<review>` block. Read that feedback. Then open `THEORY.md` and check whether the latest threat model introduces attack surfaces, invariants, or structural vulnerabilities your current script doesn't yet test — for example a new API endpoint your payload is missing, or an invariant your output hasn't attempted to break. Revise `SOLUTION.md` in place to address every reviewer point and to align with the latest threat model. Don't discard the previous script and start over, and don't leave a flagged issue unaddressed.

# f12: report back

When you finish, briefly report to the secops-director: what you did, what difficulties you hit (or didn't resolve), and any open questions.
