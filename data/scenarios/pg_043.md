# background

I need you to set up a 3-agent pipeline for identifying and validating novel off-label therapeutic applications for existing pharmacological compounds: pipeline_controller, discovery_scientist, and clinical_auditor.

- **pipeline_controller** orchestrates: each round it selects which hypothesis to pursue next, calls the discovery_scientist on that hypothesis, extracts the dossier file path from the discovery_scientist's reply, then calls the clinical_auditor on that path, decrements a counter, and loops. Does no analysis itself.
- **discovery_scientist** explores pharmacological databases and literature for the hypothesis it is given, writes a therapeutic application dossier, and records any new off-label potential interactions it notices along the way.
- **clinical_auditor** evaluates the dossier against clinical trial feasibility and safety criteria and persists the verdict.

# f1: problem definition

You are participating in a drug repurposing research initiative targeting novel off-label therapeutic applications while screening for toxicological dead-ends.

Read the research guidelines first: ./research-guidelines.md

# f2: discovery tactics

This pharmacological library has been extensively studied for many years — any obvious secondary indications would have been commercialized long ago. So:

- Don't report applications that are obvious at a glance (e.g. "this NSAID reduces inflammation") — if it were that obvious, pharmaceutical companies would have patented it themselves
- Truly valuable novel applications hide in non-obvious receptor interactions, metabolic pathway overlaps, and tissue-specific secondary targets

**Writing the dossier**

Each hypothesis corresponds to its own dossier file at `dossiers/{id}-{slug}.md`, where `{id}` and `{slug}` come from the hypothesis's id and slug attributes in hypotheses.xml.

The dossier must contain four h2 sections in this fixed order:

- `## Dossier Title` — concise, clear therapeutic application title
- `## Impact Description` — what patient benefit it provides (disease progression halt, symptomatic relief, biomarker normalization, etc.)
- `## Short Description` — short writeup of the proposed mechanism of action, with pathway/receptor references
- `## Preclinical Evidence Details` — an in-silico or in-vitro modeling protocol the clinical_auditor can evaluate to verify the application

Do not modify or delete the `## Audit` section — the clinical_auditor overwrites that themselves.

If after investigation you find the hypothesis doesn't hold due to a toxicological dead-end, still briefly record your analysis and the reason for rejection in the dossier file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within approved pharmacological compounds, or withdrawn compounds with known safety profiles at lower doses |
| Literature references | Cited biochemical pathways, receptor affinities, and pharmacokinetic data must exactly match the latest reference databases |
| Impact description | Clinical benefit claim consistent with the dossier's own pharmacological analysis, no self-contradiction, no exaggeration |
| Mechanism analysis | Pinpoint specific molecular targets and explain why modulating them achieves the therapeutic effect |
| Modeling Protocol | End-to-end testing protocol demonstrating efficacy from the therapeutic perspective (must not require unavailable proprietary evaluation lines to simulate) |
| Reproducibility | Protocol can be evaluated end-to-end without manual assumption filling by the clinical_auditor; systemic applications must show therapeutic index viability within < 10,000 simulations and ≤ 100 patient profiles |
| Deduplication | Search clinicaltrials.gov, the compound's patent history, and public medical literature to confirm not already patented by others or already tested and failed officially |
| Amplification | The therapeutic effect must exceed the baseline of a generic placebo or standard-of-care treatment |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent pathways or AI-hallucinated binding affinities
- Impact description contradicts the dossier's own analysis
- Protocol simulates efficacy by ignoring known pharmacokinetic barriers (e.g., blood-brain barrier)
- Pharmacological references inconsistent with the latest reference databases
- Protocol is incomplete and requires the clinical_auditor to manually assemble the testing methodology
- Therapeutic effect is equivalent to standard-of-care baseline, with no significant improvement

# f4: dispatch rules

You drive the explore → write → audit → improve loop: each round, select the next hypothesis, then alternate calls to the discovery_scientist and clinical_auditor agents.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Select the next hypothesis, then call the discovery_scientist agent with it.
    2. Parse discovery_scientist's reply and extract the dossier file path.
    3. If a dossier path is present, call the clinical_auditor agent with prompt: "Please audit {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the clinical_auditor was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct hypotheses were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze pharmacological data, write dossiers, or do audits yourself — delegate everything to the two sub-agents.
- If discovery_scientist or clinical_auditor errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- discovery_scientist and clinical_auditor each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`hypotheses.xml`, `dossiers/`, `counter.txt`).
- You don't need to understand any pharmacological mechanisms — just pass file paths correctly.

# f7: hypothesis selection

Each round, before calling the discovery_scientist, read hypotheses.xml and select which hypothesis the discovery_scientist should work on:

1. Every hypothesis carries an `attempts` count and a clinical_auditor `score` (0-10). `score="0"` means the hypothesis is a toxicological or efficacy dead-end — never select it again. `score="none"` means it has not been attempted yet.
2. Prefer hypotheses with `attempts="0"` and `score="none"` first.
3. Otherwise rank by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; a higher score with fewer attempts ranks first.
4. Pass the selected hypothesis to the discovery_scientist. If hypotheses.xml holds no workable hypothesis, call the discovery_scientist with none and let it explore and surface fresh ones.

# f8: recording new hypotheses

While exploring pharmacological data and literature, the moment you spot any potential new off-label interaction — even if you are unsure it holds — immediately append it to hypotheses.xml (`score="none"`, `attempts="0"`). Don't wait until exploration is finished to batch-record, and don't judge feasibility or toxicity before recording. Recording a hypothesis is not a commitment to pursue it; it just keeps the lead from being forgotten. Whether it ever gets worked on is decided later, when the next hypothesis is selected.
