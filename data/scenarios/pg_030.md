# background

I need you to set up a 3-agent pipeline for complex litigation defense drafting: docket_clerk, defense_attorney, and mock_prosecutor.

- **docket_clerk** orchestrates: calls defense_attorney, extracts the motion draft file path from defense_attorney's reply, then calls mock_prosecutor on that path, decrements a counter, and loops. Does no legal analysis itself.
- **defense_attorney** explores the case facts and statutes, picks a novel legal angle to investigate, and drafts a defense motion file.
- **mock_prosecutor** rigorously evaluates the motion for invalid citations, procedural missteps, and logical contradictions, and persists the verdict.

# f1: litigation framing

You are participating in the drafting of novel legal defense motions for high-stakes, complex litigation.

Read the court parameters and case file first: ./litigation-parameters.md

# f2: drafting tactics

The prosecution's case has been thoroughly vetted by their team—any obvious procedural errors or basic statutory violations would have been squashed long ago. So:

- Don't file motions on basic, obvious objections (e.g., "a minor typo in a subpoena")—if it were that obvious, the prosecution would have already corrected it.
- Truly valuable defense motions hide in non-obvious precedent interactions, jurisdictional overlaps, and edge-case statutory interpretations.

**Writing the motion**

Each legal angle corresponds to its own motion draft file at `motions/{id}-{slug}.md`, where `{id}` and `{slug}` come from the angle's id and slug attributes in defense_strategies.xml.

The motion draft must contain four h2 sections in this fixed order:

- `## Motion Title` — concise, clear legal filing title
- `## Strategic Impact` — what advantage it provides (evidence suppression, case dismissal, limiting witness testimony, etc.)
- `## Legal Argument` — short writeup of the defense theory, with statute/case law references
- `## Procedural Application` — a step-by-step application of the law to our specific case facts that the mock_prosecutor can evaluate as-is to test the logic

Do not modify or delete the `## Prosecutor Rebuttal` section — the mock_prosecutor overwrites that themselves.

If after investigation you find the legal angle doesn't hold, still briefly record your analysis and the reason for rejection in the motion draft file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Jurisdiction Scope | Within applicable federal/state statutes, or outside but binding via supreme court precedent |
| Case Citations | Cited case law, statute sections, and paragraph numbers must exactly match the latest LexisNexis/Westlaw database |
| Impact description | Strategic claim consistent with the motion's own legal argument, no self-contradiction, no exaggeration |
| Legal Analysis | Pinpoint specific case facts and explain why the legal theory applies |
| Procedural Application | End-to-end logical flow establishing the motion's validity from the judge's perspective (must not invent case facts to make the theory work) |
| Coherence | Argument flows without requiring the judge to make logical leaps; complex statutory defenses must hold up against all prongs of applicable legal tests |
| Deduplication | Search prior court filings, docket history, and pre-trial agreements to confirm not already argued by co-counsel or already ruled upon |
| Amplification | The strategic advantage must exceed the baseline of "a minor administrative trial delay" |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent statutes or AI-hallucinated case citations
- Impact description contradicts the motion's own legal argument
- Procedural Application invents or alters established case facts
- Case citations inconsistent with current binding precedent (e.g., overturned cases)
- Argument is incomplete and requires the judge to manually connect the dots
- Strategic advantage is merely a minor delay, with no substantive amplification of our defense

# f4: dispatch rules

Your only job is to alternate calls to the defense_attorney and mock_prosecutor agents, driving the research → draft → review → improve loop.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Call the defense_attorney agent.
    2. Parse defense_attorney's reply and extract the motion draft file path.
    3. If a motion path is present, call the mock_prosecutor agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the mock_prosecutor was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct legal angles were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze case law, draft motions, or do reviews yourself — delegate everything to the two sub-agents.
- If defense_attorney or mock_prosecutor errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- defense_attorney and mock_prosecutor each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`defense_strategies.xml`, `motions/`, `counter.txt`).
- You don't need to understand any legal arguments — just pass file paths correctly.
