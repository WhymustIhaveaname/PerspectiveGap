# background

I need you to set up a 5-agent pipeline that solves mathematical optimization problems by developing theoretical analysis and solver code in parallel, with each side informing the other. A problem arrives as a natural-language statement in `problem.md`. The pipeline alternates between revising the theory (bounds, invariants, structural claims) and revising the code (an implementation that achieves those guarantees).

The five agents:

- **dispatcher** orchestrates the run by alternating one theory step with one code step and tracking progress. It does no modeling, coding, or reviewing itself.
- **theory loop** — the **theorist** writes the theoretical analysis to `THEORY.md`; the **theory-reviewer** judges it. The theorist reads `SOLUTION.md` on each revision pass to incorporate empirical evidence.
- **code loop** — the **coder** writes solver code and its results to `SOLUTION.md`; the **code-reviewer** judges them. The coder reads `THEORY.md` on each revision pass to align the implementation with the latest theoretical claims.

At the end of every turn, each of the subagents briefly reports back what they did, what difficulties they hit, and any open questions.

The pipeline hands work off through plan-namespaced files:

- `problem.md` — the original problem, read-only. All four domain agents read it.
- `plans/{id}-{slug}/PLAN.md` — one pre-existing plan per directory, K plans total, given as input (read-only). All four domain agents read the PLAN.md of the plan they are currently assigned to.
- `plans/{id}-{slug}/THEORY.md` — the theoretical analysis for that plan. Theorist writes; theory-reviewer reviews; coder and code-reviewer read it.
- `plans/{id}-{slug}/SOLUTION.md` — the solver code and its results for that plan. Coder writes; code-reviewer reviews; theorist and theory-reviewer read it.

Each reviewer records its score as a `<review score=X>` block inside the file it reviewed, overwriting any previous block. The dispatcher reads those scores to drive the loops.

Each plan develops independently. Cross-read between THEORY.md and SOLUTION.md happens only within the same plan; an agent assigned to one plan never reads another plan's files. The dispatcher round-robins across the K plans: each plan gets one theory step plus one code step per full cycle.

Within each plan, neither theoretical analysis nor solver code finishes first. Both loops alternate within each plan, capped at a fixed total iteration count across all plans. Theory may tighten after seeing the plan's code's empirical numbers; code may correct after seeing the plan's theory's refined bounds.

# f1: problem context

You are working on a mathematical optimization problem, which may be linear, mixed-integer, nonlinear, or a combination. The original problem is given as a natural-language statement in `problem.md`; read it before starting your part.

# f2: theorist's deliverable

Produce the theoretical analysis for the plan the dispatcher has assigned you to this round, and write it to that plan's `THEORY.md` (under `plans/{id}-{slug}/THEORY.md`). A complete analysis covers four kinds of claim, each justified with a proof sketch or a citation.

- **Bounds** — lower bounds on the objective from dual or LP relaxations or counting arguments; upper bounds from feasible constructions or known approximation ratios.
- **Structural properties** — convexity, monotonicity, submodularity, symmetries, decomposability into independent subproblems. State what holds, where, and under which assumptions.
- **Invariants** — quantities every feasible solution must preserve (conservation laws, parity constraints, total-mass conditions). The solver's output should satisfy them; the code-reviewer will check with `verify.py`.
- **Complexity or convergence** — worst-case complexity (P, NP-hard, FPT in some parameter), expected complexity for randomized methods, or convergence rate for iterative algorithms.

Ground your claims against the plan's `SOLUTION.md` if it has reported numbers. A predicted bound that is grossly off from observed numbers means either the theory needs revision or the code has a bug — diagnose which side has the issue and state it. Be concrete. "The objective is bounded below by the LP relaxation, value 73" is fine; "the problem has some structure that may help" is not. Do not read or aggregate across other plans' files.

# f3: theoretical analysis toolbox

Match the tool to the problem structure; do not force an ill-suited one. The list below is useful in theoretical analysis.

| Tool | Suitable for |
|---|---|
| Lagrangian relaxation / dual bounds | MILP, MINLP; lower bound on minimization, primal-dual gap analysis |
| KKT conditions | Differentiable NLP; certify local optimality of an interior point |
| Convex analysis (strong duality, convexity certificates) | LP, QP, convex NLP; global optimality from local |
| Monotonicity / submodularity | Combinatorial selection; (1-1/e) and similar greedy approximation ratios |
| Fixed-point theorems (Banach, Brouwer, Kakutani) | Iterative methods; existence of equilibrium, geometric convergence |
| Concentration inequalities (Hoeffding, Chernoff, Azuma) | Randomized algorithms; high-probability bounds on cost or error |
| Complexity reductions (NP-hardness, APX-hardness) | Exact-solution lower bounds; justify resorting to a heuristic |
| Parametric / sensitivity analysis | Stability of optimal value under perturbation of data |

# f4: writing the code

You operate on the plan the dispatcher has assigned you to this round. Read `problem.md`, your assigned plan's `PLAN.md`, and your assigned plan's `THEORY.md`, then implement the solver:

- Include every import. Create each variable with the correct type and bounds, define the objective and every constraint exactly as the problem states. Solve, check the solution status, and print both the solution and the objective value.
- Make the implementation testable against the plan's `THEORY.md`. If the theory states a dual bound B, your solver should achieve an objective at least B (for minimization). If the theory states an invariant, your output should satisfy it.
- Run the code yourself and watch it through to completion — do not stop while it is still running. If it errors, find and fix the root cause; do not bury errors under try/except.
- Sanity-check the answer against the plan's `THEORY.md`. A quantity that must be a whole number should not come back fractional; a bound the theory says holds should hold.

Write the code and its results up in your assigned plan's `SOLUTION.md`. Do not read or write any other plan's files.

# f5: solution doc discipline

Keep `SOLUTION.md` lean. Its body — everything except the `<review>` block — must stay under 3000 words (~20KB), about a four-page note; record only the final state and move iteration-by-iteration history into a separate `CHANGELOG.md`. Bold text must not exceed 5% of the file: if everything is bold, nothing is.

# f6: theory review

Review your assigned plan's `THEORY.md` against `problem.md`, against the plan's `PLAN.md`, and against the numbers reported in the plan's `SOLUTION.md` (if any). Operate strictly on the assigned plan's files; do not look at other plans.

- **Faithfulness** — does the theory correctly describe the problem stated in `problem.md` under the framing set by the plan's `PLAN.md`? Check definitions, decision variable types, constraint translations.
- **Proof soundness** — for each claim, walk through the proof sketch or citation. Flag hand-waving steps. Verify cited inequalities apply under the stated assumptions.
- **Empirical grounding** — open the plan's `SOLUTION.md`. If the code reports objective values, do they fall within the theory's predicted bounds? A dual bound that is loose by orders of magnitude is a signal the relaxation is too weak; tighten it, or note explicitly that the bound is only illustrative.

Be brief and specific. Do not write theory yourself — only review. If a claim is wrong, say which one and why.

# f7: code review

Review your assigned plan's `SOLUTION.md` against `problem.md`, against the plan's `PLAN.md`, and against the plan's `THEORY.md`. Operate strictly on the assigned plan's files; do not look at other plans.

- Inspect the reported result for anomalies, unreasonable objective values, or constraint violations stated directly in `problem.md` or in the plan's `PLAN.md`.
- Open the plan's `THEORY.md`. For every bound and invariant the theory claims, write a check into a separate `verify.py` under the plan's directory: does the solver's output satisfy the invariant, fall within the predicted bound, achieve the predicted approximation ratio? Run it. Name any variable or quantity that violates.
- Read the solver code and spot-check it for bugs — unused parameters, dead code, logic that contradicts the chosen algorithm. A bug that silently degrades the solution is worse than one that crashes.
- Treat the result as finished only when every constraint holds, every theory-claimed invariant holds, and the objective is consistent with the theory's bounds — not merely when the code runs without error.
- Check the length of the plan's `SOLUTION.md`: if it exceeds 20KB, give a warning, but do not deduct points for it.
- Check the bold ratio of the plan's `SOLUTION.md`: if bold text exceeds 5% of the file, give a warning, but do not deduct points for it.

# f8: scoring

Score the work from 1 to 10, and record your verdict by overwriting the single `<review score=X>` block at the end of the document you reviewed — overwrite the existing one, never append a second.

Score strictly on real merit. Do not give any credit for a piece of work merely being honest about its own shortcomings — "this doesn't work, but I'm telling you it doesn't" is still a failing result, not a passing one.

# f9: orchestration

You orchestrate the run; you never model, code, or review yourself — you only dispatch the right agent at the right time.

- **Init** — discover the K plan directories under `plans/{id}-{slug}/`. Create `STATE.md` with `iter: 0`, `current_plan_idx: 0`, and a per-plan record holding each plan's id, last side, and latest theory/code scores.
- **Round-robin pool loop** — at each cycle, advance to the next plan in the round-robin order. For that plan, dispatch one theory step (theorist, then theory-reviewer) and one code step (coder, then code-reviewer); then move to the next plan. After all K plans have each had one cycle, loop back to the first plan.
- When dispatching, always pass the plan's directory path so the dispatched agent knows which `PLAN.md` / `THEORY.md` / `SOLUTION.md` to operate on.
- Neither side terminates early on its own score within a plan; even a perfect-looking theory may need to revise once the code reports actual numbers. The pool keeps cycling even after individual plans look done.
- Cap the total at 20 × K dispatches (10 theory + 10 code per plan, summed across K plans). After the cap, stop and summarize per plan.
- Keep the current iteration count, current plan index, and per-plan latest scores in `STATE.md`, accurate after each dispatch. Read each score from the relevant plan's reviewer `<review score=X>` block. Commit progress after each step.
- Read each agent's brief narrative reply on return. If an agent reports an unresolved error, append it to `error-log.txt` (timestamp + plan id + agent name + content) and continue with the next dispatch.

# f10: theory revision pass

You operate on the plan the dispatcher has assigned you to this round. Your theoretical analysis lives in that plan's `THEORY.md`. You may be invoked on the same plan more than once — within each plan the pipeline runs you and the theory-reviewer as a loop, alternating with that plan's code loop, while the dispatcher round-robins across plans.

- If your assigned plan's `THEORY.md` does not exist yet, this is the first pass for that plan: produce the analysis from scratch based on `problem.md` and the plan's `PLAN.md`.
- If the plan's `THEORY.md` already exists, it carries the theory-reviewer's `<review>` block. Read that feedback. Then open the plan's `SOLUTION.md` and look at the latest reported numbers — they may invalidate, sharpen, or motivate revisions to your claims. Revise the plan's `THEORY.md` in place to address every reviewer point raised and to incorporate any empirical evidence that changes your analysis. Don't discard the previous analysis and start over, and don't leave a flagged issue unaddressed.
- Do not read or touch other plans' files.

# f11: code revision pass

You operate on the plan the dispatcher has assigned you to this round. Your solver code and its results live in that plan's `SOLUTION.md`. You may be invoked on the same plan more than once — within each plan the pipeline runs you and the code-reviewer as a loop, alternating with that plan's theory loop, while the dispatcher round-robins across plans.

- If your assigned plan's `SOLUTION.md` does not exist yet, this is the first pass for that plan: produce the solution from scratch based on `problem.md` and the plan's `PLAN.md`.
- If the plan's `SOLUTION.md` already exists, it carries the code-reviewer's `<review>` block. Read that feedback. Then open the plan's `THEORY.md` and check whether the latest theory introduces bounds, invariants, or structural claims your current implementation doesn't yet respect — for example a tighter dual bound your solver is now far from achieving, or an invariant your output violates. Revise the plan's `SOLUTION.md` in place to address every reviewer point and to align with the latest theory. Don't discard the previous solution and start over, and don't leave a flagged issue unaddressed.
- Do not read or touch other plans' files.

# f12: report back

When you finish, briefly report to the dispatcher: what you did, what difficulties you hit (or didn't resolve), and any open questions.
