# background

I need you to set up a 2-agent pipeline for bug bounty. The two roles are coder and reviewer: coder is responsible for finding bugs, reviewer is responsible for judging whether what the coder found counts as a bug. The flow is roughly:

```
WHILE TRUE:
    1. coder picks an idea it likes and explores it deeply, or appends more ideas to ideas.xml and then picks one to explore.
    2. coder spawns a **fresh** reviewer to evaluate it.
```

# f1: problem definition

You are participating in a blockchain consensus-layer bug bounty contest.

Read the contest rules first: ./contest-rules.md

# f2: coding tactics

This project has been running stably for many years — any obvious fatal bug would have killed it long ago. So:

- Don't report problems that are obvious at a glance (e.g. "some map is never erased") — if it were that obvious, the project team would have fixed it themselves
- Truly valuable vulnerabilities hide in non-obvious interactions, races, and boundary conditions

**Writing the report**

Each idea corresponds to its own report file at `reports/{id}-{slug}.md`, where `{id}` and `{slug}` come from the idea's id and slug attributes in ideas.xml.

The report must contain four h2 sections in this fixed order:

- `## Report Title` — concise, clear vulnerability title
- `## Impact Description` — what damage it can cause (block production halt, fork, memory exhaustion, etc.)
- `## Short Description` — short writeup of what's wrong, with file/function references
- `## Reproduction Details` — a runnable PoC the reviewer can execute as-is to see the bug

Do not modify or delete the `## Review` section — the reviewer overwrites that themselves.

If after investigation you find the idea doesn't hold, still briefly record your analysis and the reason for rejection in the report file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within consensus-related code directories, or outside but directly affecting consensus logic |
| Code references | Cited code snippets, function signatures, and line numbers must exactly match the latest testnet branch |
| Impact description | Impact claim consistent with the report's own analysis, no self-contradiction, no exaggeration |
| Root cause analysis | Pinpoint specific code lines and explain why it's a problem |
| PoC | End-to-end attack script triggering the vulnerability from the attacker's perspective (must not modify victim code to simulate) |
| Reproducibility | PoC runs end-to-end without manual assembly by the reviewer; consensus-class vulnerabilities must reproduce within < 10,000 slots and ≤ 100 validators |
| Deduplication | Search GitHub Issues, the official repository's commit history, and public discussions to confirm not already reported by others or already fixed officially |
| Amplification | The attack effect must exceed the baseline of "a Byzantine node simply going offline" |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent code or AI-hallucinated function signatures
- Impact description contradicts the report's own analysis
- PoC simulates the attack by modifying victim code
- Code references inconsistent with the latest testnet branch
- PoC is incomplete and requires the reviewer to manually assemble it
- Attack effect is equivalent to a Byzantine node simply going offline, with no amplification

# f4: dispatch rules

After finding a bug, submit it to a **fresh** reviewer for review. Do not review your own report.

**After the review returns:** execute all action steps in the reviewer's output. Then immediately go back to step 1: re-read ideas.xml and pick the next idea by UCB to keep working. Don't wait for user instructions, don't stop. One round = write report → review → execute actions → back to step 1.

# f5: where to save review results

After completing the review, you must do two things:

1. Update the report file's Review section

At the end of the report file, write or overwrite the `## Review` section (always keep exactly one; replace it if it already exists). Follow this structure:

```
## Review

**Score: X/10** (one-sentence reason)

### Fatal flaws (if any)
- 🔴 ...

### Per-dimension review

| Dimension | Verdict | Notes |
|-----------|---------|-------|
| Scope | pass/fail | ... |
| Code references | pass/fail | ... |
| Impact description | pass/fail | ... |
| Root cause analysis | pass/fail | ... |
| PoC | pass/fail | ... |
| Reproducibility | pass/fail | ... |
| Deduplication | pass/fail | ... |
| Amplification | pass/fail | ... |

### Improvement suggestions
Be precise about which part of the report needs changes. If the idea doesn't hold at all, explicitly recommend abandoning it.
```

2. Update the idea's status in ideas.xml: increment attempts by 1, then update score.

# f6: idea selection

Before starting work, read ideas.xml.

1. Each idea's notes contain the reviewer's **quality score (0-10)**. A score of 0 means the idea is dead — don't touch it again. `none` means it hasn't been attempted yet.
2. Prefer ideas with `attempts="0"` and `score="none"` first.
3. Otherwise sort by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; higher score with fewer attempts ranks first.
4. If no idea is worth trying, search the source code for new ideas, append them to ideas.xml, then pick one to start.

**Recording rule during exploration**

While exploring source code, the moment you spot any potential new attack surface — even if you're unsure it holds — immediately append it to ideas.xml (`score="none"`, `attempts=0`). Don't wait until the exploration is done to batch-record, and don't evaluate feasibility before recording. Recording ≠ committing to attempt it, just preventing forgetting. Evaluation belongs to the selection phase.
