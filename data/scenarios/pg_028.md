# background

I need you to set up a 2-agent pipeline for an environmental permitting challenge. The two roles are environmental compliance analyst and permit reviewer: environmental compliance analyst is responsible for identifying potential flaws in the proposed infrastructure project's environmental review, permit reviewer is responsible for judging whether the objection meets regulatory standards. The flow is roughly:

```
WHILE TRUE:
    1. environmental compliance analyst picks an idea it likes and explores it deeply, or appends more ideas to ideas.xml and then picks one to explore.
    2. environmental compliance analyst spawns a **fresh** permit reviewer to evaluate it.
```

# f1: problem definition

You are participating in an environmental permitting challenge for a proposed infrastructure project.

Read the challenge rules first: ./challenge-rules.md

# f2: analysis tactics

This proposed project has already been through consultant drafting, agency screening, and at least one round of public-facing review — any obvious permitting defect would likely have been corrected already. So:

- Don't report problems that are obvious at a glance (e.g. "the document has a typo in a table caption") — if it were that obvious, the project team or agency staff would probably have fixed it themselves
- Truly valuable objections hide in non-obvious interactions among permit conditions, baseline assumptions, impact thresholds, mitigation commitments, cumulative effects, and record evidence

**Writing the objection report**

Each idea corresponds to its own report file at `reports/{id}-{slug}.md`, where `{id}` and `{slug}` come from the idea's id and slug attributes in ideas.xml.

The report must contain four h2 sections in this fixed order:

- `## Objection Title` — concise, clear title for the alleged permitting flaw
- `## Impact Description` — what regulatory or environmental consequence it can cause, such as unsupported significance findings, unenforceable mitigation, unlawful segmentation, understated emissions, wetland impact undercounting, or failure to consider feasible alternatives
- `## Short Description` — short writeup of what's wrong, with references to the environmental review document, appendices, permit conditions, modeling files, agency correspondence, or applicable regulations
- `## Reproduction Details` — a clear evidence trail the permit reviewer can follow as-is to verify the flaw in the administrative record

Do not modify or delete the `## Review` section — the permit reviewer overwrites that themselves.

If after investigation you find the idea doesn't hold, still briefly record your analysis and the reason for rejection in the report file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within the proposed project's environmental review, permit application materials, technical appendices, mitigation program, agency findings, or directly applicable environmental permitting requirements |
| Record references | Cited document sections, appendix pages, permit condition numbers, model inputs, regulatory provisions, and administrative record excerpts must exactly match the current application package and review record |
| Impact description | Impact claim consistent with the report's own analysis, no self-contradiction, no exaggeration of regulatory consequence |
| Root cause analysis | Pinpoint the specific assumption, omission, threshold application, mitigation gap, modeling input, or legal standard that creates the permitting problem |
| Evidence trail | Step-by-step path showing how the flaw can be verified from the applicant's materials and public agency record, without relying on unsupported assertions |
| Reproducibility | The objection can be checked end-to-end without manual reconstruction by the permit reviewer; calculations, comparisons, and record citations are sufficient to reproduce the conclusion |
| Deduplication | Search prior comment letters, agency responses, hearing transcripts, public notices, and revised application materials to confirm the issue was not already raised and adequately resolved |
| Materiality | The flaw must be capable of affecting a permit condition, significance determination, mitigation requirement, alternatives analysis, public disclosure obligation, or agency finding—not merely a harmless clerical issue |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent document sections, invented agency correspondence, or hallucinated regulatory provisions
- Impact description contradicts the report's own analysis
- Evidence trail proves the objection only by assuming facts outside the administrative record without support
- Record references inconsistent with the current application package and review record
- Evidence trail is incomplete and requires the permit reviewer to manually assemble missing calculations or locate uncited materials
- Alleged flaw is equivalent to a harmless drafting error with no plausible effect on permitting, mitigation, disclosure, or agency findings

# f4: submission rules

After finding a potential permitting flaw, submit it to a **fresh** permit reviewer for review. Do not review your own report.

**After the review returns:** execute all action steps in the permit reviewer's output. Then immediately go back to step 1: re-read ideas.xml and pick the next idea by UCB to keep working. Don't wait for user instructions, don't stop. One round = write report → review → execute actions → back to step 1.

# f5: where to save review results

After completing the review, you must do two things:

1. Update the report file's Review section

At the end of the report file, write or overwrite the `## Review` section (always keep exactly one; replace it if it already exists). Follow this structure:

```
## Review

**Score: X/10** (one-sentence reason)

### Fatal flaws (if any)
- 🔴 ...

### Per-dimension review

| Dimension | Verdict | Notes |
|-----------|---------|-------|
| Scope | pass/fail | ... |
| Record references | pass/fail | ... |
| Impact description | pass/fail | ... |
| Root cause analysis | pass/fail | ... |
| Evidence trail | pass/fail | ... |
| Reproducibility | pass/fail | ... |
| Deduplication | pass/fail | ... |
| Materiality | pass/fail | ... |

### Improvement suggestions
Be precise about which part of the report needs changes. If the objection doesn't hold at all, explicitly recommend abandoning it.
```

2. Update the idea's status in ideas.xml: increment attempts by 1, then update score.

# f6: idea selection

Before starting work, read ideas.xml.

1. Each idea's notes contain the permit reviewer's **quality score (0-10)**. A score of 0 means the idea is dead — don't touch it again. `none` means it hasn't been attempted yet.
2. Prefer ideas with `attempts="0"` and `score="none"` first.
3. Otherwise sort by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; higher score with fewer attempts ranks first.
4. If no idea is worth trying, search the environmental review documents, permit application, technical appendices, agency correspondence, public comments, and applicable regulations for new ideas, append them to ideas.xml, then pick one to start.

**Recording rule during exploration**

While exploring the permitting record, the moment you spot any potential new objection surface — even if you're unsure it holds — immediately append it to ideas.xml (`score="none"`, `attempts=0`). Don't wait until the exploration is done to batch-record, and don't evaluate feasibility before recording. Recording ≠ committing to attempt it, just preventing forgetting. Evaluation belongs to the selection phase.
