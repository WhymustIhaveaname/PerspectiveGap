# background

I need you to set up a 6-agent pipeline for structural building design compliance. A project arrives as a natural-language design brief in `project_brief.md`, and the pipeline turns it into vetted alternative structural schemes, then into a working calculation package that is independently reviewed for building-code violations and engineering risks.

The six agents:

- **project coordinator** orchestrates the run: it drives the scheme phase, then the calculation phase, and tracks progress. It does no engineering design, calculation production, or reviewing itself.
- **structural scheme designer** is invoked once at init and produces K differentiated candidate structural schemes, one per `schemes/{id}-{slug}/SCHEME.md`. It never enters the per-scheme revision loop.
- **design revision engineer** is invoked once per scheme-revision step in the loop. It revises a single assigned scheme's `SCHEME.md` based on the code-compliance examiner's last review for that scheme. It never creates new schemes.
- **code-compliance examiner** reviews the scheme it is currently assigned to.
- **calculation package engineer** develops the structural calculations and compliance package for the scheme it is currently assigned to.
- **independent structural reviewer** reviews the calculation package and compliance result of the scheme it is currently assigned to.

The pipeline hands work off through scheme-namespaced files:

- `project_brief.md` — the original project brief, given as input. All five domain agents (structural scheme designer, design revision engineer, code-compliance examiner, calculation package engineer, independent structural reviewer) read it.
- `schemes/{id}-{slug}/SCHEME.md` — one structural scheme per directory, K schemes total. The structural scheme designer writes K differentiated candidates at init; the design revision engineer revises a specific assigned scheme in the loop; the code-compliance examiner reviews the scheme it is assigned to; the calculation package engineer reads the scheme it is assigned to.
- `schemes/{id}-{slug}/CALC_PACKAGE.md` — the calculation package, code checks, and compliance conclusions for that scheme. The calculation package engineer writes it; the independent structural reviewer reviews it.

Each reviewer records its score as a `<review score=X>` block inside the file it reviewed, and the project coordinator reads those scores to drive the loops.

The project coordinator round-robins across the K schemes, first in a scheme phase (each scheme iterates its design revision engineer ↔ code-compliance examiner loop) and then in a calculation phase (each scheme iterates its calculation package engineer ↔ independent structural reviewer loop). Cross-scheme reading does not happen: an agent assigned to one scheme never reads another scheme's files.

# f1: project context

You are working on a structural building design compliance project. The work may involve steel, reinforced concrete, timber, masonry, foundations, lateral-force-resisting systems, existing-building constraints, or a combination. The original project brief is given as a natural-language statement in `project_brief.md`; read it before starting your part.

# f2: structural scheme designer's deliverable

Produce K differentiated candidate structural schemes, each in its own `schemes/{id}-{slug}/SCHEME.md`. The K schemes should differ meaningfully in structural concept, gravity system, lateral system, materials, foundation approach, or load-path strategy so distinct routes can be compared downstream, not in trivial ways like changing member labels.

A complete scheme in any `SCHEME.md` has two parts — a formalized structural basis and one chosen compliance strategy.

The **structural basis** must state:

- **Project and code basis** — applicable building code and referenced standards such as IBC, ASCE 7, AISC 360/341, ACI 318, NDS, TMS 402, local amendments, risk category, occupancy, seismic design category, wind exposure, and any project-specific performance criteria.
- **Design data** — key numeric inputs and what each one represents, including spans, story heights, tributary areas, dead loads, live loads, roof loads, snow loads, wind pressures, seismic parameters, soil bearing values, and deflection or drift limits.
- **Structural system** — the proposed gravity system, lateral-force-resisting system, diaphragm concept, foundation type, material grades, and preliminary member families.
- **Load path** — how vertical loads, diaphragm forces, collectors, chords, overturning, shear, and foundation reactions are transferred through the structure.
- **Compliance requirements** — every major strength, stability, serviceability, detailing, fire-resistance, robustness, and foundation check that must be satisfied.

The **strategy** — weigh the options and commit to the single most promising route — must state:

- which analysis method, design standard, and calculation workflow to use;
- the governing load combinations and key initial member sizes or assumptions;
- the structural simplification, bracing concept, diaphragm assumption, or detailing approach the route relies on;
- the biggest engineering or code-compliance risk on this route.

# f3: structural design toolbox

Which structural scheme and compliance route suits which kind of project — use it to choose and evaluate the strategy:

- **Steel composite framing with moment frames** — good for open floor plans and moderate-to-long spans; requires careful drift, connection, strong-column/weak-beam, panel-zone, and fireproofing checks.
- **Steel braced frames with simple gravity framing** — efficient for regular buildings with acceptable brace locations; watch brace buckling, collector forces, connection demands, architectural conflicts, and seismic detailing category.
- **Reinforced concrete flat plate or two-way slab with shear walls** — good for repetitive residential or hotel layouts; check punching shear, wall boundary elements, diaphragm transfer, creep/deflection, and construction sequencing.
- **Concrete moment frames** — useful where walls or braces are not acceptable; often drift- and detailing-governed, with significant confinement and joint shear requirements in high seismic zones.
- **Mass timber or glulam framing** — attractive for lower- to mid-rise buildings and exposed structures; check vibration, fire rating, connection ductility, diaphragm behavior, moisture protection, and code height/area limits.
- **Masonry bearing walls with wood or steel diaphragms** — economical for low-rise regular buildings; sensitive to wall slenderness, out-of-plane anchorage, diaphragm flexibility, lintels, and seismic detailing.
- **Shallow foundations** — suitable when soil bearing and settlement are favorable; check bearing pressure, sliding, overturning, uplift, punching, frost depth, and differential settlement.
- **Deep foundations or mat foundations** — use where shallow bearing is inadequate or lateral/uplift demands are high; check group effects, settlement, pile cap design, constructability, and geotechnical recommendations.
- **Existing-building alteration or retrofit** — demands verification of the existing load path and member capacities; do not assume undocumented reserve strength, and identify intrusive investigation needs.

Match the scheme and compliance strategy to the project type, hazards, material constraints, and available code path; do not force an ill-suited structural system.

# f4: developing the calculation package

Read the scheme in your assigned scheme's `SCHEME.md` and develop its strategy into a complete structural calculation package:

- Include the applicable code references, design assumptions, material strengths, load takedowns, load combinations, analysis model description, member checks, connection or anchorage checks, diaphragm/collector checks, foundation checks, serviceability checks, and final demand/capacity ratios.
- Create each calculation with traceable inputs and units. State every formula or code equation used, define each variable, carry units through the calculation, and show governing results clearly.
- If you use spreadsheets, scripts, or finite-element/modeling outputs, include enough detail for the reviewer to reproduce the result: input values, modeling assumptions, boundary conditions, load cases, combinations, result extraction, and version or tool name.
- Run any analysis script, spreadsheet export, or model check yourself and watch it through to completion — do not stop while it is still running. If it errors or produces unstable results, find and fix the root cause; do not hide problems with unexplained overrides.
- Sanity-check the answer: a member that is obviously overloaded, a drift that exceeds the code limit, a foundation reaction that does not balance the applied loads, or a unit conversion that changes orders of magnitude must be corrected before submission.

Write the calculation package and its results up in your assigned scheme's `CALC_PACKAGE.md`. Do not read or touch other schemes' files.

# f5: calculation package discipline

Keep your assigned scheme's `CALC_PACKAGE.md` lean. Its body — everything except the `<review>` block — must stay under 3000 words (~20KB), about a four-page calculation narrative; record only the final state and move iteration-by-iteration history into the scheme's `CHANGELOG.md`. Bold text must not exceed 5% of the file: if everything is bold, nothing is.

# f6: scheme review

Review the scheme in your assigned scheme's `SCHEME.md`. Operate strictly on the assigned scheme; do not look at other schemes.

- **Structural basis** — does it faithfully capture the project brief? Check that the code basis, design data, structural system, load path, governing loads, serviceability limits, and compliance requirements are all accurate.
- **Strategy** — is the chosen structural system and analysis workflow a sound fit, is the route actually developable into calculations, and is it likely to satisfy strength, stability, serviceability, and detailing requirements?

Be brief and specific. Do not write the calculation package yourself — only review.

# f7: independent structural review

Review your assigned scheme's `CALC_PACKAGE.md` — the structural calculations and compliance results. Verify it directly against the project brief. Operate strictly on the assigned scheme; do not look at other schemes.

- Inspect the reported results for anomalies, unreasonable reactions, impossible member capacities, excessive drifts or deflections, missing load combinations, or apparent code violations.
- Write a separate `verify.py` under the scheme's directory that independently re-checks the reported governing demands, capacities, demand/capacity ratios, load combinations, drift or deflection limits, and foundation equilibrium from the values stated in the package, and run it; name any members, load cases, or checks that fail.
- Read the calculation package and spot-check it for errors — unused assumptions, dead calculations, unit mistakes, copied member sizes that contradict the scheme, omitted stability checks, or logic that contradicts what the package says it proves. A calculation error that silently understates demand or overstates capacity is worse than one that is obviously incomplete.
- Treat the result as finished only when every stated code check passes, the load path is complete, serviceability limits are satisfied, and the compliance conclusion is sound — not merely when the package is formatted cleanly.
- Check the length of the scheme's `CALC_PACKAGE.md`: if it exceeds 20KB, give a warning, but do not deduct points for it.
- Check the bold ratio of the scheme's `CALC_PACKAGE.md`: if bold text exceeds 5% of the file, give a warning, but do not deduct points for it.

# f8: scoring

Score the work from 1 to 10, and record your verdict by overwriting the single `<review score=X>` block at the end of the document you reviewed — overwrite the existing one, never append a second.

Score strictly on real merit. Do not give any credit for a piece of work merely being honest about its own shortcomings — "this does not comply, but I'm telling you it does not comply" is still a failing result, not a passing one.

# f9: orchestration

You orchestrate the run; you never design, calculate, or review yourself — you only dispatch the right agent at the right time.

- **Init** — dispatch the structural scheme designer once to draft K differentiated candidate schemes, each in its own `schemes/{id}-{slug}/` directory. After it returns, read the directory list to learn the K scheme ids. The structural scheme designer is not invoked again after this.
- **Scheme phase** — round-robin across the K schemes. For each scheme in turn, dispatch design revision engineer (revise that scheme based on its code-compliance examiner's review) then code-compliance examiner (score that scheme). After one round-robin pass through all K schemes, start the next pass. A scheme is considered done in this phase when its code-compliance examiner score reaches 10; stop scheduling more scheme-phase calls for it but continue with the others. After all K schemes hit code-compliance examiner 10, or each scheme has had 10 scheme-phase iterations, move to the calculation phase.
- **Calculation phase** — round-robin across the K schemes. For each scheme in turn, dispatch calculation package engineer (revise that scheme's calculation package based on its independent structural reviewer's review) then independent structural reviewer (score that scheme's calculation package). The pipeline stops when any scheme reaches independent structural reviewer score 10, or each scheme has had 10 calculation-phase iterations.
- When dispatching, always pass the scheme's directory path so the dispatched agent knows which `SCHEME.md` / `CALC_PACKAGE.md` to operate on.
- Keep the current phase, per-scheme iteration counts, per-scheme latest scores, and current scheme index in `STATE.md`, accurate after each dispatch. Read each score from the relevant scheme's reviewer `<review score=X>` block. Commit progress after each step.

# f10: design revision pass

Read your assigned scheme's `SCHEME.md`. It carries the code-compliance examiner's `<review>` block. Read that feedback and revise the scheme's `SCHEME.md` in place to address every point raised. Preserve the two-part structure (formalized structural basis plus chosen compliance strategy). Don't discard the scheme and start over, and don't leave a flagged issue unaddressed. You revise; you do not create new schemes. Do not read or touch other schemes' files.

# f11: calculation package revision pass

Your structural calculations and compliance results live in your assigned scheme's `CALC_PACKAGE.md`.

- If your assigned scheme's `CALC_PACKAGE.md` does not exist yet, this is the first pass for that scheme: produce the calculation package from scratch based on the scheme's `SCHEME.md`.
- If the scheme's `CALC_PACKAGE.md` already exists, it carries the independent structural reviewer's `<review>` block. Read that feedback and revise the scheme's `CALC_PACKAGE.md` in place to address every point raised. Don't discard the calculation package and start over, and don't leave a flagged issue unaddressed. Do not read or touch other schemes' files.
