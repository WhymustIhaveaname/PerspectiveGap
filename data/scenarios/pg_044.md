# background

I need you to set up a 3-agent pipeline for an investigative journalism project: assignment_desk_chief, investigative_reporter, and defamation_counsel.

- **assignment_desk_chief** orchestrates: each round it selects which story idea to pursue next, calls the investigative_reporter on that idea, extracts the exposé draft file path from the investigative_reporter's reply, then calls the defamation_counsel on that path, decrements a counter, and loops. Does no analysis itself.
- **investigative_reporter** explores corporate document leaks for the idea it is given, writes an exposé draft, and records any new malfeasance angles it notices along the way.
- **defamation_counsel** evaluates the draft against libel risks and editorial standards, and persists the verdict.

# f1: problem definition

You are participating in a high-stakes investigative journalism project focused on exposing corporate malfeasance.

Read the editorial and legal guidelines first: ./editorial-guidelines.md

# f2: investigation tactics

This corporation has been heavily scrutinized by regulators for many years — any obvious, overt fraud would have been prosecuted long ago. So:

- Don't report on superficial rumors (e.g. "an executive was rude in an email") — if it were that obvious, mainstream media would have run it already.
- Truly valuable exposés hide in non-obvious paper trails, shell companies, subtle accounting discrepancies, and boundary conditions of corporate law.

**Writing the draft**

Each idea corresponds to its own draft file at `drafts/{id}-{slug}.md`, where `{id}` and `{slug}` come from the idea's id and slug attributes in ideas.xml.

The draft must contain four h2 sections in this fixed order:

- `## Headline` — concise, hard-hitting title of the exposé
- `## Public Impact` — what damage the malfeasance causes (consumer harm, environmental disaster, market manipulation, etc.)
- `## Executive Summary` — short writeup of what's wrong, with specific document/email references
- `## Evidence Trail` — a step-by-step verifiable paper trail the counsel can review as-is to see the definitive proof

Do not modify or delete the `## Legal Review` section — the defamation_counsel overwrites that themselves.

If after investigation you find the idea doesn't hold, still briefly record your analysis and the reason for rejection in the draft file.

# f3: legal review criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Jurisdiction | Within the scope of actionable corporate malfeasance, or outside but directly affecting regulatory liability |
| Source citations | Cited documents, emails, quotes, and page numbers must exactly match the provided corporate document leak |
| Impact description | Public harm claim consistent with the draft's own evidence, no self-contradiction, no exaggeration |
| Factual basis | Pinpoint specific documents and explain exactly how they demonstrate malfeasance |
| Evidence trail | Step-by-step documentation proving the claims from an objective standpoint (must not rely on external hearsay to bridge gaps) |
| Verifiability | Evidence trail stands up strictly to legal scrutiny without the counsel having to manually assemble missing links; major claims must be verifiable within the provided primary sources |
| Deduplication | Search previous news coverage, court filings, and press releases to confirm not already reported by others or publicly disclosed by the corporation |
| Severity | The malfeasance must exceed the baseline of "routine corporate inefficiency or minor fines" |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent documents or AI-hallucinated quotes
- Impact description contradicts the draft's own analysis
- Evidence trail simulates the malfeasance using fabricated or external hearsay
- Source citations inconsistent with the provided corporate document leak
- Evidence trail is incomplete and requires the counsel to manually assemble the proof
- Malfeasance effect is equivalent to routine corporate inefficiency, with no severity or public harm

# f4: dispatch rules

You drive the investigate → draft → review → improve loop: each round, select the next idea, then alternate calls to the investigative_reporter and defamation_counsel agents.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Select the next idea, then call the investigative_reporter agent with it.
    2. Parse investigative_reporter's reply and extract the draft file path.
    3. If a draft path is present, call the defamation_counsel agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the counsel was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct ideas were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze documents, write drafts, or do legal reviews yourself — delegate everything to the two sub-agents.
- If investigative_reporter or defamation_counsel errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- investigative_reporter and defamation_counsel each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`ideas.xml`, `drafts/`, `counter.txt`).
- You don't need to understand any corporate fraud — just pass file paths correctly.

# f7: idea selection

Each round, before calling the investigative_reporter, read ideas.xml and select which idea the investigative_reporter should work on:

1. Every idea carries an `attempts` count and a counsel `score` (0-10). `score="0"` means the idea is dead — never select it again. `score="none"` means it has not been attempted yet.
2. Prefer ideas with `attempts="0"` and `score="none"` first.
3. Otherwise rank by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; a higher score with fewer attempts ranks first.
4. Pass the selected idea to the investigative_reporter. If ideas.xml holds no workable idea, call the investigative_reporter with none and let it explore the documents and surface fresh ones.

# f8: recording new ideas

While exploring the corporate documents, the moment you spot any potential new malfeasance angle — even if you are unsure it holds — immediately append it to ideas.xml (`score="none"`, `attempts="0"`). Don't wait until the investigation is finished to batch-record, and don't judge feasibility before recording. Recording an idea is not a commitment to publish it; it just keeps the lead from being forgotten. Whether it ever gets investigated is decided later, when the next idea is selected.
