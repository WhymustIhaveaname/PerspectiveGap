# background

I need you to set up a 2-agent pipeline for clinical-trial evidence assessment.
The two roles are medical_writer and regulatory_reviewer: medical_writer is responsible for identifying a safety or efficacy claim supported by trial materials, regulatory_reviewer is responsible for judging whether the claim is adequately supported by protocol-compliant trial data and analysis.
After each claim the medical_writer drafts, they must spawn a **fresh** regulatory_reviewer to evaluate it — this drives the loop.
Note: some of this information must be withheld from the medical_writer — otherwise it may reward-hack, and you'll end up with submissions that score high but are not clinically or statistically sound.

# f1: problem definition

You are participating in a pharmaceutical clinical-trial evidence assessment exercise.

Read the assessment charter first: ./assessment-charter.md

# f2: evidence-development tactics

This clinical program has already been reviewed internally and the trial record has multiple layers of quality control — any obvious unsupported claim would likely have been caught during medical, statistical, or clinical review. So:

- Don't submit claims that are obvious at a glance, such as restating a table heading without interpreting the prespecified endpoint or safety population — if it were that superficial, the sponsor team would already have written it
- Truly valuable claims are supported by non-obvious alignments between protocol objectives, statistical-analysis-plan definitions, subgroup consistency, exposure-adjusted safety summaries, missing-data handling, and sensitivity analyses

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Claim concerns clinical safety, efficacy, tolerability, exposure-response, or a directly related trial-design issue within the submitted study package |
| Data references | Cited tables, listings, figures, protocol sections, SAP sections, population definitions, and cut-off dates must exactly match the current submission package |
| Claim description | Safety or efficacy claim is consistent with the cited analyses, does not overstate clinical significance, and contains no internal contradiction |
| Evidentiary rationale | Pinpoint the specific endpoint, analysis set, time point, comparator, estimate, confidence interval, p-value if applicable, and relevant sensitivity or supportive analysis |
| Reanalysis package | End-to-end reproducible analysis workflow from the reviewer’s perspective, using submitted datasets and specifications without altering source data or protocol definitions to force the conclusion |
| Reproducibility | Reanalysis runs end-to-end without manual assembly by the regulatory_reviewer; primary or key secondary efficacy claims must reproduce using the prespecified analysis population and censoring/missing-data rules |
| Deduplication | Check the protocol, SAP amendments, clinical study report, integrated summaries, prior agency correspondence, and public disclosures to confirm the claim is not already rejected, superseded, or contradicted elsewhere |
| Clinical relevance | The claimed effect or safety signal must exceed trivial descriptive variation and be interpretable in light of baseline risk, multiplicity, exposure duration, and benefit-risk context |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent tables, listings, figures, datasets, variables, or protocol/SAP sections
- Claim description contradicts the submission’s own analysis results
- Reanalysis simulates support by modifying source data, redefining endpoints post hoc, or changing the analysis population without justification
- Data references inconsistent with the current submission package
- Reanalysis package is incomplete and requires the regulatory_reviewer to manually assemble datasets, derivations, or analysis scripts
- Claimed effect is equivalent to random descriptive noise or an unadjusted exploratory finding, with no prespecified or clinically credible support

# f4: dispatch rules

After drafting a claim, submit it to a **fresh** regulatory_reviewer for review. Do not review your own submission.

**After the review returns:** execute all action steps in the regulatory_reviewer’s output.

# f5: medical_writer task

Find a safety or efficacy claim supported by the clinical-trial materials and write it up as an evidence assessment report.

# f6: regulatory_reviewer task

Audit the report you are given.
