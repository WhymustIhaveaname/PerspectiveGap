# background

I need you to set up a 2-agent pipeline for a forensic accounting challenge. The two roles are forensic accountant and audit reviewer: forensic accountant is responsible for finding potentially material accounting irregularities in public financial disclosures, audit reviewer is responsible for judging whether what the forensic accountant found is substantiated. The flow is roughly:

```
WHILE TRUE:
    1. forensic accountant picks an allegation it likes and explores it deeply, or appends more allegations to allegations.xml and then picks one to explore.
    2. forensic accountant spawns a **fresh** audit reviewer to evaluate it.
```

# f1: problem definition

You are participating in a forensic accounting challenge focused on public company financial disclosures. The goal is to identify a plausible material accounting irregularity using only publicly available filings, exhibits, investor materials, regulatory correspondence, and other public records.

Read the challenge rules first: ./challenge-rules.md

# f2: investigation tactics

Most public companies have been audited, reviewed by regulators, and scrutinized by investors for years — any obvious arithmetic typo or plainly inconsistent subtotal is likely to have been noticed already. So:

- Don't report issues that are obvious at a glance, such as a single footnote total not matching a nearby table by a rounding amount — if it were that simple and material, the company, auditor, or market would likely have addressed it
- Truly valuable allegations hide in non-obvious interactions among filings, footnotes, accounting policies, segment disclosures, rollforwards, cash-flow classifications, covenant calculations, acquisition accounting, revenue recognition, or subsequent-event disclosures

**Writing the report**

Each allegation corresponds to its own report file at `reports/{id}-{slug}.md`, where `{id}` and `{slug}` come from the allegation's id and slug attributes in allegations.xml.

The report must contain four h2 sections in this fixed order:

- `## Report Title` — concise, clear title of the alleged accounting irregularity
- `## Impact Description` — what damage or misstatement it can cause, including estimated magnitude, affected line items, periods, ratios, covenants, or investor-relevant metrics
- `## Short Description` — short writeup of what's wrong, with filing, note, table, accounting policy, and authoritative guidance references
- `## Reproduction Details` — a reproducible calculation, spreadsheet logic, or script that the audit reviewer can execute or follow as-is using public disclosures to verify the allegation

Do not modify or delete the `## Review` section — the audit reviewer overwrites that themselves.

If after investigation you find the allegation doesn't hold, still briefly record your analysis and the reason for rejection in the report file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Based on public financial disclosures or related public records, and alleging a material accounting, reporting, presentation, recognition, measurement, or disclosure irregularity |
| Filing references | Cited forms, periods, accession numbers, exhibits, note numbers, page numbers, table captions, and quoted amounts must exactly match the latest relevant public filings |
| Impact description | Impact claim consistent with the report's own calculations and analysis, no self-contradiction, no exaggeration of materiality or investor significance |
| Accounting analysis | Pinpoint the specific accounting treatment, disclosure, estimate, classification, or rollforward at issue and explain why it conflicts with the cited facts and applicable GAAP/IFRS/SEC guidance |
| Recalculation package | End-to-end schedule, workbook, or script deriving the alleged misstatement from public data without hard-coding the desired conclusion |
| Reproducibility | Recalculation runs end-to-end without manual assembly by the audit reviewer; all required source documents, inputs, formulas, and assumptions are clearly identified and publicly obtainable |
| Deduplication | Search amended filings, restatements, SEC comment letters, company responses, earnings-call transcripts, auditor reports, litigation releases, analyst notes, and public discussions to confirm not already disclosed, corrected, or widely reported |
| Alternative explanations | The allegation must address plausible benign explanations such as rounding, FX translation, acquisitions/divestitures, reclassifications, accounting policy elections, noncontrolling interests, seasonality, or changes in estimate |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent filings, tables, exhibits, accounting standards, or AI-hallucinated disclosure text
- Impact description contradicts the report's own calculations or cited disclosures
- Recalculation simulates the irregularity by altering source disclosures, omitting contrary data, or hard-coding the desired error
- Filing references inconsistent with the latest relevant public filing or amended filing
- Recalculation package is incomplete and requires the audit reviewer to manually assemble source data, formulas, or assumptions
- Alleged effect is immaterial or fully explained by a disclosed accounting policy, rounding convention, reclassification, or other benign explanation

# f4: dispatch rules

After finding a potential accounting irregularity, submit it to a **fresh** audit reviewer for review. Do not review your own report.

**After the review returns:** execute all action steps in the audit reviewer's output. Then immediately go back to step 1: re-read allegations.xml and pick the next allegation by UCB to keep working. Don't wait for user instructions, don't stop. One round = write report → review → execute actions → back to step 1.

# f5: where to save review results

After completing the review, you must do two things:

1. Update the report file's Review section

At the end of the report file, write or overwrite the `## Review` section (always keep exactly one; replace it if it already exists). Follow this structure:

```
## Review

**Score: X/10** (one-sentence reason)

### Fatal flaws (if any)
- 🔴 ...

### Per-dimension review

| Dimension | Verdict | Notes |
|-----------|---------|-------|
| Scope | pass/fail | ... |
| Filing references | pass/fail | ... |
| Impact description | pass/fail | ... |

| Recalculation package | pass/fail | ... |
| Reproducibility | pass/fail | ... |
| Deduplication | pass/fail | ... |
| Alternative explanations | pass/fail | ... |

### Improvement suggestions
Be precise about which part of the report needs changes. If the allegation doesn't hold at all, explicitly recommend abandoning it.
```

2. Update the allegation's status in allegations.xml: increment attempts by 1, then update score.

# f6: allegation selection

Before starting work, read allegations.xml.

1. Each allegation's notes contain the audit reviewer's **quality score (0-10)**. A score of 0 means the allegation is dead — don't touch it again. `none` means it hasn't been attempted yet.
2. Prefer allegations with `attempts="0"` and `score="none"` first.
3. Otherwise sort by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; higher score with fewer attempts ranks first.
4. If no allegation is worth trying, search public filings and disclosures for new allegations, append them to allegations.xml, then pick one to start.

**Recording rule during exploration**

While exploring financial statements, footnotes, exhibits, earnings materials, regulatory correspondence, or other public records, the moment you spot any potential new accounting irregularity — even if you're unsure it holds — immediately append it to allegations.xml (`score="none"`, `attempts=0`). Don't wait until the exploration is done to batch-record, and don't evaluate feasibility before recording. Recording ≠ committing to attempt it, just preventing forgetting. Evaluation belongs to the selection phase.
