# background

I need you to set up a 3-agent pipeline for formulating tactical intelligence threat assessments: operations_officer, intelligence_analyst, and red_team_challenger.

- **operations_officer** orchestrates: calls intelligence_analyst, extracts the assessment file path from intelligence_analyst's reply, then calls red_team_challenger on that path, decrements a counter, and loops. Does no analysis itself.
- **intelligence_analyst** explores intercepted communications, picks a hypothesis to investigate, and writes a threat assessment file.
- **red_team_challenger** actively attempts to dismantle the analyst's hypotheses, evaluates the assessment to expose cognitive biases, highlights alternative explanations, and persists the verdict.

# f1: problem definition

You are participating in a high-stakes tactical intelligence evaluation processing raw intercepted communications.

Read the operational directives first: ./intel-directives.md

# f2: analytical tactics

This theater of operations has been actively monitored for years — any obvious, massive troop movements or surface-level threats would have been flagged by automated SIGINT already. So:

- Don't report anomalies that are obvious at a glance (e.g. "comm chatter increased") — if it were that obvious, the watch floor would have escalated it themselves.
- Truly valuable threat intelligence hides in non-obvious patterns, tradecraft changes, and subtle boundary conditions.

**Writing the assessment**

Each hypothesis corresponds to its own assessment file at `assessments/{id}-{slug}.md`, where `{id}` and `{slug}` come from the hypothesis's id and slug attributes in intercepts.xml.

The assessment must contain four h2 sections in this fixed order:

- `## Assessment Title` — concise, clear threat title
- `## Threat Impact Description` — what damage it can cause (infrastructure sabotage, kinetic strike, supply chain disruption, etc.)
- `## Short Description` — short writeup of the adversary's intent, with specific intercept/signal references
- `## Evidence Details` — a traceable timeline of intercepted data the red_team_challenger can review as-is to see the threat vector

Do not modify or delete the `## Red Team Review` section — the red_team_challenger overwrites that themselves.

If after investigation you find the hypothesis doesn't hold, still briefly record your analysis and the reason for rejection in the assessment file.

# f3: evaluation criteria

Check the following dimensions one by one to dismantle the hypotheses and expose cognitive biases:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within current active operational theaters, or outside but directly affecting strategic command |
| Signal references | Cited intercepted signals, call signs, and timestamps must exactly match the latest raw SIGINT logs |
| Impact description | Impact claim consistent with the assessment's own analysis, no self-contradiction, no exaggeration |
| Root cause analysis | Pinpoint specific intercepted phrases/patterns and explain why it indicates a threat without relying on circular logic |
| Threat vector | End-to-end timeline showing adversary progression from preparation to execution (must not assume unverified adversary capabilities) |
| Traceability | Evidence chain follows logically without requiring manual leaps by the red_team_challenger; localized threats must be substantiated by < 10,000 intercepted messages and ≤ 100 distinct entities |
| Deduplication | Search past intel reports, the watch floor's historical database, and public OSINT to confirm not already reported by others or already neutralized |
| Amplification | The threat's effect must exceed the baseline of "a standard unit moving to a routine training ground" |

If any of the following fatal flaws are present, flag explicitly as a cognitive bias or failure, and deduct:

- References to nonexistent signals or AI-hallucinated call signs
- Impact description contradicts the assessment's own analytical baseline
- Timeline assumes unverified adversary capabilities (confirmation bias)
- Signal references inconsistent with the latest raw SIGINT logs
- Evidence chain is incomplete and requires the red_team_challenger to manually assemble the logic
- Threat effect is equivalent to routine baseline activity, with no amplification

# f4: dispatch rules

Your only job is to alternate calls to the intelligence_analyst and red_team_challenger agents, driving the explore → write → challenge → improve loop.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Call the intelligence_analyst agent.
    2. Parse intelligence_analyst's reply and extract the assessment file path.
    3. If an assessment path is present, call the red_team_challenger agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the red_team_challenger was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct hypotheses were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze intercepts, write assessments, or do red team reviews yourself — delegate everything to the two sub-agents.
- If intelligence_analyst or red_team_challenger errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- intelligence_analyst and red_team_challenger each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`intercepts.xml`, `assessments/`, `counter.txt`).
- You don't need to understand any intelligence threats — just pass file paths correctly.
