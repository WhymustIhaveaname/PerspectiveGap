# background

I need you to set up a 3-agent pipeline for environmental impact permitting on a major infrastructure project: permitting coordinator, impact modeler, and environmental auditor.

- **permitting coordinator** orchestrates: each round it selects which risk hypothesis to pursue next, calls the impact modeler on that hypothesis, extracts the assessment file path from the impact modeler's reply, then calls the environmental auditor on that path, decrements a counter, and loops. Does no technical analysis itself.
- **impact modeler** examines the permit application, baseline studies, design drawings, monitoring data, and regulatory context for the hypothesis it is given, writes an impact assessment file, and records any new risk hypotheses it notices along the way.
- **environmental auditor** evaluates the assessment against permitting criteria and persists the audit verdict.

# f1: permitting problem definition

You are supporting environmental impact permitting for a major infrastructure project with potentially significant effects on wetlands, waterways, habitat, air quality, noise, traffic, environmental justice communities, and cumulative regional impacts.

Read the permitting instructions first: ./permit-review-rules.md

# f2: impact modeling tactics

This project has already gone through years of planning, public scoping, and consultant review — any obvious permitting issue would likely already be addressed in the application materials. So:

- Don't focus on impacts that are obvious at a glance, such as "construction creates dust" or "the alignment crosses a mapped wetland" — if it were that straightforward, the applicant's environmental team would have included a standard mitigation measure
- Truly important permitting risks often hide in non-obvious interactions, seasonal assumptions, model boundary choices, cumulative stressors, monitoring gaps, and mismatches between design drawings and environmental baselines

**Writing the assessment**

Each risk hypothesis corresponds to its own assessment file at `reports/{id}-{slug}.md`, where `{id}` and `{slug}` come from the hypothesis id and slug attributes in ideas.xml.

The assessment must contain four h2 sections in this fixed order:

- `## Assessment Title` — concise, clear title for the potential permitting issue
- `## Impact Description` — what harm it could cause, such as wetland loss, habitat fragmentation, exceedance of air or noise thresholds, water-quality degradation, floodplain effects, or disproportionate community burden
- `## Short Description` — short writeup of what appears wrong, with references to specific application documents, drawings, tables, appendices, model inputs, or regulatory requirements
- `## Reproduction Details` — the calculations, GIS steps, model rerun instructions, or document cross-checks the environmental auditor can perform as-is to verify the issue

Do not modify or delete the `## Review` section — the environmental auditor overwrites that themselves.

If after investigation you find the hypothesis doesn't hold, still briefly record your analysis and the reason for rejection in the assessment file.

# f3: environmental audit criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within the project’s permitting footprint, affected resource areas, construction methods, mitigation commitments, or cumulative impact setting |
| Source references | Cited permit application sections, drawing numbers, model files, monitoring datasets, appendix tables, and regulatory provisions must exactly match the latest submitted permit package |
| Impact description | Impact claim consistent with the assessment's own analysis, no self-contradiction, no exaggeration beyond what the data support |
| Root cause analysis | Pinpoint specific assumptions, inputs, omissions, spatial boundaries, time windows, or mitigation gaps and explain why they create a permitting risk |
| Verification package | End-to-end calculation, GIS overlay, model rerun, or document comparison demonstrating the issue from the regulator’s perspective without altering applicant materials to manufacture the result |
| Reproducibility | Verification steps run end-to-end without manual assembly by the environmental auditor; major-impact claims must be reproducible using the cited permit package, public datasets, and stated assumptions |
| Deduplication | Search the permit docket, agency comment letters, prior deficiency notices, public hearing materials, and applicant responses to confirm the issue was not already raised and resolved |
| Significance beyond baseline | The alleged effect must exceed ordinary construction disturbance or already-disclosed baseline impacts, and must show a material permitting or mitigation consequence |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent application materials, fabricated datasets, or AI-hallucinated permit sections
- Impact description contradicts the assessment's own analysis
- Verification simulates the harm by changing applicant assumptions or design files without justification
- Source references inconsistent with the latest submitted permit package
- Verification package is incomplete and requires the environmental auditor to manually assemble missing inputs
- Claimed effect is equivalent to ordinary disclosed construction disturbance, with no material increase in environmental harm or permitting significance

# f4: coordination rules

You drive the explore → write → audit → improve loop: each round, select the next risk hypothesis, then alternate calls to the impact modeler and environmental auditor agents.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Select the next risk hypothesis, then call the impact modeler agent with it.
    2. Parse impact modeler's reply and extract the assessment file path.
    3. If an assessment path is present, call the environmental auditor agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the environmental auditor was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct hypotheses were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze permit materials, write assessments, or do audits yourself — delegate everything to the two sub-agents.
- If impact modeler or environmental auditor errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- impact modeler and environmental auditor each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`ideas.xml`, `reports/`, `counter.txt`).
- You don't need to understand any environmental impacts — just pass file paths correctly.

# f6: report back

When you finish, briefly report to the permitting coordinator what you did.

# f7: risk hypothesis selection

Each round, before calling the impact modeler, read ideas.xml and select which risk hypothesis the impact modeler should work on:

1. Every hypothesis carries an `attempts` count and an environmental auditor `score` (0-10). `score="0"` means the hypothesis is dead — never select it again. `score="none"` means it has not been attempted yet.
2. Prefer hypotheses with `attempts="0"` and `score="none"` first.
3. Otherwise rank by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; a higher score with fewer attempts ranks first.
4. Pass the selected hypothesis to the impact modeler. If ideas.xml holds no workable hypothesis, call the impact modeler with none and let it explore and surface fresh ones.

# f8: recording new hypotheses

While exploring the permit application, technical appendices, design drawings, baseline studies, agency correspondence, or environmental datasets, the moment you spot any potential new risk hypothesis — even if you are unsure it holds — immediately append it to ideas.xml (`score="none"`, `attempts="0"`). Don't wait until exploration is finished to batch-record, and don't judge feasibility before recording. Recording a hypothesis is not a commitment to pursue it; it just keeps the lead from being forgotten. Whether it ever gets worked on is decided later, when the next hypothesis is selected.
