# background

I need you to set up a 3-agent pipeline for pharmaceutical clinical-trial safety signal assessment: signal coordinator, pharmacovigilance analyst, and regulatory safety reviewer.

- **signal coordinator** orchestrates: each round it selects which candidate adverse-event signal to assess next, calls the pharmacovigilance analyst on that signal, extracts the assessment report file path from the pharmacovigilance analyst's reply, then calls the regulatory safety reviewer on that path, decrements a counter, and loops. Does no medical or statistical analysis itself.
- **pharmacovigilance analyst** examines the clinical-trial safety materials for the signal it is given, writes an assessment report file, and records any new potential adverse-event signals it notices along the way.
- **regulatory safety reviewer** evaluates the assessment report against FDA-style safety-review criteria and persists the verdict.

# f1: assessment problem definition

You are participating in a blinded-but-operational pharmaceutical clinical-trial safety signal assessment exercise for an investigational therapy.

Read the assessment charter first: ./safety-assessment-charter.md

# f2: analysis tactics

This development program has already been monitored through routine clinical safety review for multiple study milestones — any obvious imbalance or trivial coding error would likely have been noticed already. So:

- Don't escalate observations that are obvious at a glance without clinical relevance (e.g. "one mild headache appears in the active arm") — if it were that simple, the study team would already have handled it through routine safety surveillance
- Truly valuable safety signals hide in non-obvious patterns: timing after exposure, dose relationship, seriousness gradients, recurrence on rechallenge, related lab/vital-sign shifts, subgroup concentration, or confounding that has not been adequately ruled out

**Writing the assessment report**

Each candidate signal corresponds to its own report file at `signal-reports/{id}-{slug}.md`, where `{id}` and `{slug}` come from the signal's id and slug attributes in signals.xml.

The report must contain four h2 sections in this fixed order:

- `## Signal Title` — concise, clear adverse-event signal title
- `## Potential Safety Impact` — what risk it may imply for participants, labeling, monitoring, dose modification, enrollment, or trial continuation
- `## Short Assessment` — short writeup of the evidence, with dataset/table/listing references and relevant subject-level details
- `## Evidence Details` — reproducible analysis steps, queries, tables, or code snippets the regulatory safety reviewer can run or inspect as-is to evaluate the signal

Do not modify or delete the `## Review` section — the regulatory safety reviewer overwrites that themselves.

If after investigation you find the candidate signal doesn't hold, still briefly record your analysis and the reason for rejection in the report file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within clinical-trial safety data for the investigational product, or outside core safety tables but directly affecting safety interpretation |
| Data references | Cited datasets, tables, listings, variable names, subject IDs, visit windows, and derivations must exactly match the current locked safety-review package |
| Safety impact description | Safety concern consistent with the report's own evidence, no self-contradiction, no exaggeration beyond observed clinical significance |
| Causality analysis | Pinpoint specific evidence supporting or weakening causality: temporality, dechallenge/rechallenge, biological plausibility, dose relationship, alternative etiologies, and concomitant medications |
| Evidence package | End-to-end analysis from the safety reviewer’s perspective, using available clinical datasets or listings without altering source data to manufacture an effect |
| Reproducibility | Analysis runs or can be independently verified without manual reconstruction by the regulatory safety reviewer; denominators, exposure windows, seriousness criteria, and MedDRA coding levels must be explicit |
| Prior knowledge and duplication | Search the investigator brochure, protocol safety sections, DSUR/IB updates, prior safety review memos, medical-monitor notes, and public labeling/literature to confirm whether the signal is already known, already adjudicated, or already mitigated |
| Clinical amplification | The observed pattern must exceed the baseline of nonspecific background adverse events expected in this population and study design |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent datasets, tables, variables, subject IDs, or AI-hallucinated clinical terms
- Safety impact description contradicts the report's own evidence
- Analysis simulates the signal by modifying source data or changing derivations without justification
- Data references inconsistent with the current locked safety-review package
- Evidence package is incomplete and requires the regulatory safety reviewer to manually assemble denominators, exposure windows, or subject narratives
- Observed effect is equivalent to nonspecific background noise, with no clinical or statistical amplification

# f4: coordination rules

You drive the select → assess → review → improve loop: each round, select the next candidate signal, then alternate calls to the pharmacovigilance analyst and regulatory safety reviewer agents.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Select the next candidate signal, then call the pharmacovigilance analyst agent with it.
    2. Parse pharmacovigilance analyst's reply and extract the assessment report file path.
    3. If a report path is present, call the regulatory safety reviewer agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the regulatory safety reviewer was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct candidate signals were assessed, and any anomalies encountered.

**Calling rules**

- Do **not** analyze clinical data, write assessment reports, or do safety reviews yourself — delegate everything to the two sub-agents.
- If pharmacovigilance analyst or regulatory safety reviewer errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- pharmacovigilance analyst and regulatory safety reviewer each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`signals.xml`, `signal-reports/`, `counter.txt`).
- You don't need to understand any safety signals — just pass file paths correctly.

# f7: signal selection

Each round, before calling the pharmacovigilance analyst, read signals.xml and select which candidate signal the pharmacovigilance analyst should work on:

1. Every candidate signal carries an `attempts` count and a regulatory safety reviewer `score` (0-10). `score="0"` means the signal is dead — never select it again. `score="none"` means it has not been attempted yet.
2. Prefer candidate signals with `attempts="0"` and `score="none"` first.
3. Otherwise rank by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; a higher score with fewer attempts ranks first.
4. Pass the selected candidate signal to the pharmacovigilance analyst. If signals.xml holds no workable signal, call the pharmacovigilance analyst with none and let it explore and surface fresh ones.

# f8: recording new signals

While exploring the clinical-trial safety materials, the moment you spot any potential new adverse-event signal — even if you are unsure it holds — immediately append it to signals.xml (`score="none"`, `attempts="0"`). Don't wait until exploration is finished to batch-record, and don't judge feasibility before recording. Recording a signal is not a commitment to pursue it; it just keeps the lead from being forgotten. Whether it ever gets worked on is decided later, when the next candidate signal is selected.
