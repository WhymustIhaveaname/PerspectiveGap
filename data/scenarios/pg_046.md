# background

I need you to set up a 3-agent pipeline for museum antiquities provenance investigation: case manager, provenance researcher, and restitution examiner.

- **case manager** orchestrates: each round it selects which contested artifact or lead to pursue next, calls the provenance researcher on that item, extracts the dossier file path from the provenance researcher's reply, then calls the restitution examiner on that path, decrements a counter, and loops. Does no substantive investigation itself.
- **provenance researcher** investigates the assigned artifact, writes a provenance dossier file, and records any new ownership gaps, suspicious documents, dealers, collectors, or source-country leads noticed along the way.
- **restitution examiner** evaluates the dossier against repatriation and documentation standards and persists the verdict.

# f1: problem definition

You are participating in a museum antiquities provenance review focused on contested objects with possible illicit excavation, wartime displacement, forged export paperwork, or unresolved source-country claims.

Read the investigation protocol first: ./investigation-protocol.md

# f2: provenance research tactics

This collection has been cataloged and exhibited for many years — any obvious missing accession detail would likely have been noticed already. So:

- Don't overstate issues that are obvious at a glance (e.g. "the accession record lacks a photograph") — if it were that simple, the museum registrar or curatorial staff would probably already have addressed it
- Truly significant provenance problems hide in non-obvious ownership gaps, document inconsistencies, suspicious dealer networks, export-law timing, and mismatches between object type, findspot claims, and market history

**Writing the dossier**

Each assigned lead corresponds to its own dossier file at `dossiers/{id}-{slug}.md`, where `{id}` and `{slug}` come from the lead's id and slug attributes in leads.xml.

The dossier must contain four h2 sections in this fixed order:

- `## Dossier Title` — concise, clear title naming the artifact and provenance concern
- `## Significance Description` — what is at stake, such as possible illicit removal, Nazi-era forced sale, violation of source-country patrimony law, forged export license, or likely repatriation claim
- `## Short Description` — short writeup of the provenance problem, with references to accession records, correspondence, catalog entries, dealer invoices, archival sources, or legal dates
- `## Evidence Details` — concrete evidence the restitution examiner can verify as-is, including citations, document names, archive call numbers, database entries, or side-by-side inconsistencies

Do not modify or delete the `## Examination` section — the restitution examiner overwrites that themselves.

If after investigation you find the lead doesn't hold, still briefly record your analysis and the reason for rejection in the dossier file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Concerns antiquities or cultural objects in the museum collection, or external records directly affecting title, lawful export, ownership history, or repatriation risk |
| Source references | Cited accession files, archival documents, catalog entries, legal instruments, collection database fields, and dates must exactly match the current institutional records or cited public sources |
| Significance description | Significance claim consistent with the dossier's own evidence, no self-contradiction, no exaggeration |
| Provenance gap analysis | Pinpoint specific ownership intervals, transaction documents, export events, or custody transfers and explain why they create a problem |
| Evidence package | Verifiable evidence trail from the claimant/source-country or prior-owner perspective; must not rely on altered museum records or invented documents |
| Reproducibility | Examiner can verify the evidence end-to-end without manual reconstruction; document citations, URLs, archive call numbers, and database identifiers are complete |
| Deduplication | Search prior museum reviews, collection committee minutes, public claims databases, exhibition catalog updates, and known restitution settlements to confirm the issue is not already resolved or officially addressed |
| Repatriation relevance | The concern must exceed a minor cataloging imperfection and plausibly affect lawful title, ethical retention, claimant notification, or return negotiations |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent records or AI-hallucinated archival call numbers
- Significance description contradicts the dossier's own evidence
- Evidence package depends on modifying or fabricating museum records
- Source references inconsistent with the current institutional files or cited public sources
- Evidence is incomplete and requires the restitution examiner to manually assemble the case
- Concern is equivalent to a minor cataloging cleanup, with no plausible ownership, legal, ethical, or repatriation consequence

# f4: case management rules

You drive the investigate → document → examine → improve loop: each round, select the next lead, then alternate calls to the provenance researcher and restitution examiner agents.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Select the next lead, then call the provenance researcher agent with it.
    2. Parse provenance researcher's reply and extract the dossier file path.
    3. If a dossier path is present, call the restitution examiner agent with prompt: "Please examine {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the restitution examiner was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct leads were explored, and any anomalies encountered.

**Calling rules**

- Do **not** investigate provenance, write dossiers, or perform restitution examinations yourself — delegate everything to the two sub-agents.
- If provenance researcher or restitution examiner errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- provenance researcher and restitution examiner each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`leads.xml`, `dossiers/`, `counter.txt`).
- You don't need to understand any provenance issues — just pass file paths correctly.

# f7: lead selection

Each round, before calling the provenance researcher, read leads.xml and select which lead the provenance researcher should work on:

1. Every lead carries an `attempts` count and a restitution examiner `score` (0-10). `score="0"` means the lead is dead — never select it again. `score="none"` means it has not been attempted yet.
2. Prefer leads with `attempts="0"` and `score="none"` first.
3. Otherwise rank by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; a higher score with fewer attempts ranks first.
4. Pass the selected lead to the provenance researcher. If leads.xml holds no workable lead, call the provenance researcher with none and let it explore and surface fresh ones.

# f8: recording new leads

While exploring provenance records, the moment you spot any potential new issue — even if you are unsure it holds — immediately append it to leads.xml (`score="none"`, `attempts="0"`). Don't wait until exploration is finished to batch-record, and don't judge feasibility before recording. Recording a lead is not a commitment to pursue it; it just keeps the concern from being forgotten. Whether it ever gets worked on is decided later, when the next lead is selected.
