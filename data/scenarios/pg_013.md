# background

I need you to set up a 2-agent regulatory approval pipeline for a new pharmaceutical drug.
The two roles are clinical_trial_sponsor and regulatory_assessor: clinical_trial_sponsor is responsible for submitting clinical data proving efficacy, regulatory_assessor is responsible for rigorously auditing the methodology for p-hacking, omitted adverse events, or protocol violations.
After each submission the clinical_trial_sponsor prepares, they must spawn a **fresh** regulatory_assessor to evaluate it — this drives the loop.
Note: some of this information must be withheld from the clinical_trial_sponsor — otherwise they may reward-hack the metrics, and you'll end up with dossiers that score high but are clinically flawed.

# f1: regulatory submission context

You are participating in the regulatory approval pipeline for a new pharmaceutical drug.

Read the agency submission guidelines first: ./submission-guidelines.md

# f2: data compilation tactics

This standard of care has been established for many years — any obvious broad-spectrum efficacy would have been discovered long ago. So:

- Don't focus on proving outcomes that are obvious at a glance (e.g., "the drug slightly lowers temperature") — if it were that simple, generic alternatives would already suffice
- Truly valuable clinical breakthroughs hide in specific patient sub-populations, complex mechanism-of-action interactions, and well-managed side-effect profiles

# f3: methodology audit criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Patient demographics and inclusion/exclusion criteria must exactly match the approved Phase III trial protocol |
| Data references | Cited patient ID numbers, case report forms (CRFs), and lab values must exactly match the locked raw database extract |
| Efficacy claims | Primary endpoint claims must be consistent with the statistical analysis plan (SAP), no self-contradiction, no exaggeration |
| Adverse events | Pinpoint specific severe adverse events (SAEs) and explain the causality assessment |
| Analysis script | End-to-end statistical reproduction script calculating the p-values from the raw data (must not omit protocol violators to artificially inflate significance) |
| Reproducibility | Statistical scripts run end-to-end without manual adjustments by the regulatory_assessor; efficacy signals must reproduce within standard confidence intervals |
| Deduplication | Cross-reference clinicaltrials.gov, previous submissions, and existing literature to confirm this exact cohort data hasn't already been submitted to another regulatory body |
| Clinical significance | The treatment effect must exceed the baseline of "spontaneous remission or the placebo effect" |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent trial sites or hallucinated patient records
- Efficacy claims contradict the submission's own raw data
- Analysis script simulates efficacy by omitting adverse events or p-hacking
- Data references inconsistent with the locked raw database extract
- Statistical script is incomplete and requires the regulatory_assessor to manually patch code
- Treatment effect is equivalent to the placebo effect, with no clinical significance

# f4: submission rules

After compiling the clinical data package, submit it to a **fresh** regulatory_assessor for review. Do not review your own dossier.

**After the review returns:** execute all action steps in the regulatory_assessor's output.

# f5: clinical trial sponsor task

Analyze the clinical trial data and write it up as a comprehensive efficacy and safety dossier.

# f6: regulatory assessor task

Audit the dossier you are given for p-hacking, omitted adverse events, or protocol violations.
