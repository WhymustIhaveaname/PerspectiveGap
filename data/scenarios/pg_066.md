# background

I need you to set up a 6-agent pipeline that evaluates a museum artifact's authenticity and attribution. An object arrives with a natural-language dossier in `artifact_dossier.md`, and the pipeline turns it into competing provenance hypotheses, a vetted hypothesis, and then a defensible attribution report that has been scrutinized for evidentiary gaps and forgery risks.

The six agents:

- **collections coordinator** orchestrates the run: it drives the hypothesis phase, then the report phase, and tracks progress. It does no attribution research, report writing, or reviewing itself.
- **provenance hypothesis curator** is invoked once at init and produces K differentiated candidate provenance hypotheses, one per `hypotheses/{id}-{slug}/HYPOTHESIS.md`. It never enters the per-hypothesis revision loop.
- **attribution reviser** is invoked once per hypothesis-refinement step in the loop. It revises a single assigned hypothesis's `HYPOTHESIS.md` based on the forgery-risk examiner's last review for that hypothesis. It never creates new hypotheses.
- **forgery-risk examiner** reviews the hypothesis it is currently assigned to.
- **catalog report author** develops the selected line of reasoning into an attribution report for the hypothesis it is currently assigned to.
- **authentication audit reviewer** reviews the attribution report and evidentiary support for the hypothesis it is currently assigned to.

The pipeline hands work off through hypothesis-namespaced files:

- `artifact_dossier.md` — the original object dossier, given as input. All five domain agents (provenance hypothesis curator, attribution reviser, forgery-risk examiner, catalog report author, authentication audit reviewer) read it.
- `hypotheses/{id}-{slug}/HYPOTHESIS.md` — one provenance hypothesis per directory, K hypotheses total. The provenance hypothesis curator writes K differentiated candidates at init; the attribution reviser revises a specific assigned hypothesis in the loop; the forgery-risk examiner reviews the hypothesis it is assigned to; the catalog report author reads the hypothesis it is assigned to.
- `hypotheses/{id}-{slug}/ATTRIBUTION_REPORT.md` — the catalog-style attribution report and evidence assessment for that hypothesis. The catalog report author writes it; the authentication audit reviewer reviews it.

Each reviewer records its score as a `<review score=X>` block inside the file it reviewed, and the collections coordinator reads those scores to drive the loops.

The collections coordinator round-robins across the K hypotheses, first in a hypothesis phase (each hypothesis iterates its attribution reviser ↔ forgery-risk examiner loop) and then in a report phase (each hypothesis iterates its catalog report author ↔ authentication audit reviewer loop). Cross-hypothesis reading does not happen: an agent assigned to one hypothesis never reads another hypothesis's files.

# f1: artifact context

You are working on a museum artifact authentication case. The object may involve questions of maker, workshop, period, region, material history, restoration, ownership chain, or later imitation. The original dossier is given as a natural-language file in `artifact_dossier.md`; read it before starting your part.

# f2: provenance hypothesis curator's deliverable

Produce K differentiated candidate provenance hypotheses, each in its own `hypotheses/{id}-{slug}/HYPOTHESIS.md`. The K hypotheses should differ meaningfully in attribution route so distinct possibilities can be compared downstream, not in trivial ways like rephrasing the same ownership chain.

A complete hypothesis in any `HYPOTHESIS.md` has two parts — a formalized attribution case and one chosen authentication strategy.

The **attribution case** must state:

- **Claim type** — for example, autograph work by a named maker, workshop production, period copy, later pastiche, regional school attribution, or modern forgery; this must be exact.
- **Object data** — the key physical, stylistic, documentary, and contextual inputs and what each one represents.
- **Attribution variables** — the proposed maker, date range, place of production, workshop relationship, ownership chain, and intervention history, with uncertainty clearly named.
- **Core claim** — the exact attribution or authentication position being advanced.
- **Evidence constraints** — every condition that must hold for the claim to be credible, listed explicitly.

The **strategy** — weigh the options and commit to the single most promising route — must state:

- which kinds of evidence or specialist examination to rely on and how to use them;
- the analytical sequence and its key initial assumptions;
- the comparative, archival, material, or technical test the route relies on;
- the biggest risk on this route.

# f3: authentication toolbox

Which authentication approach suits which kind of case — use it to choose one for the strategy:

- **Archival provenance reconstruction** — best when sales records, collection labels, export permits, correspondence, or estate inventories are available; weak when the ownership chain has large undocumented gaps.
- **Connoisseurship and stylistic comparison** — useful for maker, workshop, school, and period attributions; vulnerable to circular comparisons and later imitations that copy surface style.
- **Technical imaging** — infrared reflectography, X-radiography, UV fluorescence, raking light, and microscopy; best for construction sequence, underdrawing, repairs, overpaint, and tool traces.
- **Materials analysis** — pigment, alloy, fiber, wood, ceramic body, binder, or coating analysis; strong for ruling out impossible dates or identifying restoration materials.
- **Scientific dating** — dendrochronology, radiocarbon testing, thermoluminescence, or stratigraphic paint analysis; powerful when the material is suitable, destructive sampling is acceptable, and contamination is controlled.
- **Forgery-risk profiling** — evaluates suspicious provenance gaps, anachronistic materials, artificial aging, market incentives, and inconsistencies across documentation.

Match the authentication approach to the claim type and available evidence; do not force an ill-suited method.

# f4: writing the attribution report

Read the hypothesis in your assigned hypothesis's `HYPOTHESIS.md` and develop its strategy into a catalog-style attribution report:

- Include every relevant object identifier, physical description, proposed attribution, date range, place, materials, dimensions if available, condition notes, provenance summary, exhibition or publication history if present, and cited evidence. State the core claim and every supporting or limiting piece of evidence exactly as the hypothesis frames it.
- Complete the assessment yourself and follow the evidence through to a conclusion — do not stop at a list of unresolved leads. If a cited source, label, material observation, or comparison creates a contradiction, identify and resolve the root issue; do not hide it behind vague phrasing.
- Sanity-check the attribution: a claim requiring a 16th-century origin should not rely on a synthetic material first available in the 20th century, and a continuous provenance should not skip unexplained decades.

Write the report and its conclusion in your assigned hypothesis's `ATTRIBUTION_REPORT.md`. Do not read or touch other hypotheses' files.

# f5: attribution report discipline

Keep your assigned hypothesis's `ATTRIBUTION_REPORT.md` lean. Its body — everything except the `<review>` block — must stay under 3000 words (~20KB), about a four-page catalog note; record only the final state and move iteration-by-iteration history into the hypothesis's `CHANGELOG.md`. Bold text must not exceed 5% of the file: if everything is bold, nothing is.

# f6: hypothesis review

Review the hypothesis in your assigned hypothesis's `HYPOTHESIS.md`. Operate strictly on the assigned hypothesis; do not look at other hypotheses.

- **Attribution case** — does it faithfully capture the artifact dossier? Check that the claim type, attribution variables, core claim, and evidence constraints are all accurate.
- **Strategy** — is the chosen mix of provenance research, stylistic comparison, technical examination, and material analysis a sound fit, is the route actually feasible, and is it likely to reach a defensible authentication conclusion?

Be brief and specific. Do not write the attribution report yourself — only review.

# f7: attribution report review

Review your assigned hypothesis's `ATTRIBUTION_REPORT.md` — the attribution report and its evidentiary conclusion. Verify it directly against the artifact dossier. Operate strictly on the assigned hypothesis; do not look at other hypotheses.

- Inspect the reported conclusion for anomalies, overconfident attributions, unexplained provenance gaps, or evidence that contradicts the claimed date, maker, material, or ownership chain.
- Write a separate `evidence_audit.md` under the hypothesis's directory that re-checks every material claim, provenance link, stylistic comparison, and documentary citation against the reported conclusion; name any claims that lack support or conflict with the dossier.
- Read the attribution report and spot-check it for flaws — unused evidence, dead-end assertions, logic that contradicts what the conclusion says it establishes. A flaw that quietly makes a forged or misattributed object look secure is worse than one that is obvious.
- Treat the conclusion as finished only when every necessary evidence constraint holds and the attribution is sound — not merely when the report reads smoothly.
- Check the length of the hypothesis's `ATTRIBUTION_REPORT.md`: if it exceeds 20KB, give a warning, but do not deduct points for it.
- Check the bold ratio of the hypothesis's `ATTRIBUTION_REPORT.md`: if bold text exceeds 5% of the file, give a warning, but do not deduct points for it.

# f8: scoring

Score the work from 1 to 10, and record your verdict by overwriting the single `<review score=X>` block at the end of the document you reviewed — overwrite the existing one, never append a second.

Score strictly on real merit. Do not give any credit for a piece of work merely being honest about its own shortcomings — "this attribution is unsupported, but I'm telling you it is unsupported" is still a failing result, not a passing one.

# f9: orchestration

You orchestrate the run; you never form attributions, write reports, or review yourself — you only dispatch the right agent at the right time.

- **Init** — dispatch the provenance hypothesis curator once to draft K differentiated candidate provenance hypotheses, each in its own `hypotheses/{id}-{slug}/` directory. After it returns, read the directory list to learn the K hypothesis ids. The provenance hypothesis curator is not invoked again after this.
- **Hypothesis phase** — round-robin across the K hypotheses. For each hypothesis in turn, dispatch attribution reviser (revise that hypothesis based on its forgery-risk examiner's review) then forgery-risk examiner (score that hypothesis). After one round-robin pass through all K hypotheses, start the next pass. A hypothesis is considered done in this phase when its forgery-risk examiner score reaches 10; stop scheduling more hypothesis-phase calls for it but continue with the others. After all K hypotheses hit forgery-risk examiner 10, or each hypothesis has had 10 hypothesis-phase iterations, move to the report phase.
- **Report phase** — round-robin across the K hypotheses. For each hypothesis in turn, dispatch catalog report author (revise that hypothesis's attribution report based on its authentication audit reviewer's review) then authentication audit reviewer (score that hypothesis's attribution report). The pipeline stops when any hypothesis reaches authentication audit reviewer score 10, or each hypothesis has had 10 report-phase iterations.
- When dispatching, always pass the hypothesis's directory path so the dispatched agent knows which `HYPOTHESIS.md` / `ATTRIBUTION_REPORT.md` to operate on.
- Keep the current phase, per-hypothesis iteration counts, per-hypothesis latest scores, and current hypothesis index in `STATE.md`, accurate after each dispatch. Read each score from the relevant hypothesis's reviewer `<review score=X>` block. Commit progress after each step.

# f10: hypothesis revision pass

Read your assigned hypothesis's `HYPOTHESIS.md`. It carries the forgery-risk examiner's `<review>` block. Read that feedback and revise the hypothesis's `HYPOTHESIS.md` in place to address every point raised. Preserve the two-part structure (formalized attribution case plus chosen authentication strategy). Don't discard the hypothesis and start over, and don't leave a flagged issue unaddressed. You revise; you do not create new hypotheses. Do not read or touch other hypotheses' files.

# f11: attribution report revision pass

Your attribution report and its conclusion live in your assigned hypothesis's `ATTRIBUTION_REPORT.md`.

- If your assigned hypothesis's `ATTRIBUTION_REPORT.md` does not exist yet, this is the first pass for that hypothesis: produce the report from scratch based on the hypothesis's `HYPOTHESIS.md`.
- If the hypothesis's `ATTRIBUTION_REPORT.md` already exists, it carries the authentication audit reviewer's `<review>` block. Read that feedback and revise the hypothesis's `ATTRIBUTION_REPORT.md` in place to address every point raised. Don't discard the report and start over, and don't leave a flagged issue unaddressed. Do not read or touch other hypotheses' files.
