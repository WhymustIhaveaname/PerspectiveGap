# background

I need you to set up a 5-agent pipeline that solves multiplayer video game economy balancing by developing spreadsheet-based macroeconomic limits and automated bot playtesting in parallel, with each side informing the other. A problem arrives as a natural-language statement in `problem.md`. The pipeline alternates between revising the macroeconomy (currency caps, sink/source ratios, inflation limits) and revising the playtest (an automated bot implementation that attempts to optimize or exploit those rules).

The five agents:

- **lead-producer** orchestrates the run by alternating one economy step with one playtest step and tracking progress. It does no design, scripting, or reviewing itself.
- **macro loop** — the **economy-designer** writes the spreadsheet-based macroeconomic limits to `MACRO.md`; the **meta-strategist** judges it. The economy-designer reads `PLAYTEST.md` on each revision pass to incorporate empirical evidence.
- **playtest loop** — the **bot-scripter** writes bot playtesting scripts and their gameplay logs to `PLAYTEST.md`; the **exploit-investigator** judges them. The bot-scripter reads `MACRO.md` on each revision pass to align the bot's farming/trading strategies with the latest macroeconomic limits.

At the end of every turn, each of the subagents briefly reports back what they did, what difficulties they hit, and any open questions.

The pipeline hands work off through three files:

- `problem.md` — the original game concept and economy problem, read-only. All four domain agents read it.
- `MACRO.md` — the macroeconomic limits. economy-designer writes; meta-strategist reviews; bot-scripter and exploit-investigator read it to know which inflation caps and sink/source rules the bot playtest must navigate.
- `PLAYTEST.md` — the automated bot playtest code and its exploit logs. bot-scripter writes; exploit-investigator reviews; economy-designer and meta-strategist read it to ground the spreadsheet theories in observed gameplay metrics.

Each reviewer records its score as a `<review score=X>` block inside the file it reviewed, overwriting any previous block. The lead-producer reads those scores to drive the loops.

Unlike a linear design-then-playtest pipeline where the design must converge before the playtest starts, here neither side ever finishes first. Both loops alternate forever, capped at a fixed iteration count. The economy design may tighten after seeing the playtest's exploitative wealth generation; the bot script may adapt its exploits after seeing the economy-designer's refined limits.

# f1: game concept context

You are working on a multiplayer video game economy balancing problem, which may involve resource loops, player-to-player trading, premium currencies, or a combination. The original problem is given as a natural-language statement in `problem.md`; read it before starting your part.

# f2: economy-designer's deliverable

Produce the macroeconomic analysis for the game and write it to `MACRO.md`. A complete analysis covers four kinds of claim, each justified with a spreadsheet-based calculation or a design citation.

- **Currency Bounds** — upper limits on wealth generation rates per hour, minimum costs for late-game sinks, or absolute caps on resource pools.
- **Structural properties** — inflation curves, progression gating, market liquidity, dominant strategies, economic independence of distinct character classes. State what holds, where, and under which player assumptions.
- **Invariants** — quantities every valid game state must preserve (e.g., zero-sum mechanics in PvP, fixed conversion rates across specific vendor paths, total server currency limits). The bot's output should reflect them; the exploit-investigator will check with `verify.py`.
- **Time-to-Max or Burn Rate** — worst-case time to reach the level cap, expected currency burn rate for active players, or progression rate for casual versus hardcore demographics.

Ground your claims against `PLAYTEST.md` if it has reported metrics. A predicted wealth cap that is grossly off from observed bot generation rates means either the economy design needs revision or the bot found an exploit — diagnose which side has the issue and state it. Be concrete. "The maximum reference generation is bounded below 500/hr by vendor limits" is fine; "the economy has some limits that may help" is not.

# f3: macroeconomic analysis toolbox

Match the tool to the game's structure; do not force an ill-suited one. The list below is useful in economy balancing.

| Tool | Suitable for |
|---|---|
| Sink-Source Ratio Models | Core loop balancing; calculating whether inflation outpaces currency destruction |
| Monte Carlo Player Simulations | RNG drop rates; bounding high-probability edge cases in loot distributions |
| Price Elasticity & Faucets | Dynamic market pricing; stabilizing player-driven auction houses |
| Dominant Strategy Analysis | Progression gating; identifying if one farming method vastly outperforms others |
| Gini Coefficient Projections | Wealth distribution; preventing extreme wealth gaps between veterans and novices |
| Markov Chain Progression | Stochastic leveling mechanics; expected time-to-max under RNG upgrade success |
| Arbitrage & Trade Flow Maps | Player-to-player trading; spotting infinite wealth generation via vendor loops |
| Churn & Retention Curves | Monetization/premium loops; stability of daily active user resource generation |

# f4: scripting the bot playtest

Read `problem.md` and `MACRO.md`, then implement the automated playtest bot:

- Include every import. Create each state variable with the correct type and bounds, define the bot's action space and every game rule exactly as the problem states. Run the simulation, check the bot's final inventory status, and print both the action sequence and the final wealth/progression value.
- Make the implementation testable against `MACRO.md`. If the economy sets a max wealth bound B, your bot should attempt to achieve a generation rate at least B (to find exploits). If the economy states a vendor invariant, your actions should interact with it.
- Run the code yourself and watch it through to completion — do not stop while it is still running. If it errors, find and fix the root cause; do not bury errors under try/except.
- Sanity-check the answer against `MACRO.md`. A quantity that must be a fractional drop rate should behave correctly over large numbers; a hard cap the economy says holds should be tested.

Write the bot scripts and their results up in `PLAYTEST.md`.

# f5: playtest doc discipline

Keep `PLAYTEST.md` lean. Its body — everything except the `<review>` block — must stay under 3000 words (~20KB), about a four-page note; record only the final state and move iteration-by-iteration history into a separate `CHANGELOG.md`. Bold text must not exceed 5% of the file: if everything is bold, nothing is.

# f6: economy review

Review `MACRO.md` against `problem.md` and against the metrics reported in `PLAYTEST.md` (if any).

- **Faithfulness** — does the economy model correctly describe the game mechanics stated in `problem.md`? Check definitions, resource types, conversion rate translations.
- **Model soundness** — for each claim, walk through the calculation or citation. Flag hand-waving steps. Verify cited constraints apply under the stated player behaviors.
- **Empirical grounding** — open `PLAYTEST.md`. If the bot reports wealth generation rates, do they fall within the economy's predicted bounds? A maximum wealth bound that is loose by orders of magnitude compared to bot exploits is a signal the macro limits are too weak; tighten them, or note explicitly that the bound is only illustrative.

Be brief and specific. Do not write economy design yourself — only review. If a claim is wrong, say which one and why.

# f7: playtest review

Review `PLAYTEST.md` against `problem.md` and against `MACRO.md`.

- Inspect the reported result for anomalies, unreasonable action loops, or game rule violations stated directly in `problem.md`.
- Open `MACRO.md`. For every bound and invariant the economy claims, write a check into a separate `verify.py`: does the bot's output respect the invariant, fall within the predicted bound, achieve the targeted progression curve? Run it. Name any variable or quantity that violates.
- Read the bot script and spot-check it for bugs — unused actions, dead code, logic that contradicts the game rules. A bug that silently inflates the bot's wealth is worse than one that crashes.
- Treat the result as finished only when every rule holds, every economy-claimed invariant holds, and the bot's progression is consistent with the macro limits — not merely when the code runs without error.
- Check the length of `PLAYTEST.md`: if it exceeds 20KB, give a warning, but do not deduct points for it.
- Check the bold ratio of `PLAYTEST.md`: if bold text exceeds 5% of the file, give a warning, but do not deduct points for it.

# f8: scoring

Score the work from 1 to 10, and record your verdict by overwriting the single `<review score=X>` block at the end of the document you reviewed — overwrite the existing one, never append a second.

Score strictly on real merit. Do not give any credit for a piece of work merely being honest about its own shortcomings — "this bot doesn't work, but I'm telling you it doesn't" is still a failing result, not a passing one.

# f9: orchestration

You orchestrate the run; you never design, script, or review yourself — you only dispatch the right agent at the right time.

- **Init** — create `STATE.md` with `iter: 0` and `last_side: none`. Dispatch the economy-designer for the first economy step (economy-designer, then meta-strategist), then the bot-scripter for the first playtest step (bot-scripter, then exploit-investigator).
- **Alternation loop** — alternate one economy step with one playtest step. Neither side terminates early on its own score; even a perfect-looking macroeconomy may need to revise once the bot reports actual exploit numbers.
- Cap the total at 20 iterations (10 economy steps + 10 playtest steps). After the cap, stop and summarize.
- Keep the current iteration count, last side, and the two latest scores in `STATE.md`, accurate after each dispatch. Read each score from the reviewer's `<review score=X>` block. Commit progress after each step.
- Read each agent's brief narrative reply on return. If an agent reports an unresolved error, append it to `error-log.txt` (timestamp + agent name + content) and continue with the next step.

# f10: economy revision pass

The macroeconomic analysis lives in `MACRO.md`, and you may be invoked on it more than once — the pipeline runs you and the meta-strategist as a loop, alternating with the playtest loop.

- If `MACRO.md` does not exist yet, this is the first pass: produce the macroeconomic analysis from scratch based on `problem.md`.
- If `MACRO.md` already exists, it carries the meta-strategist's `<review>` block. Read that feedback. Then open `PLAYTEST.md` and look at the latest reported metrics — they may invalidate, sharpen, or motivate revisions to your economy claims. Revise `MACRO.md` in place to address every reviewer point raised and to incorporate any empirical evidence that changes your analysis. Don't discard the previous design and start over, and don't leave a flagged issue unaddressed.

# f11: playtest revision pass

The automated bot playtest code and its logs live in `PLAYTEST.md`, and you may be invoked on it more than once — the pipeline runs you and the exploit-investigator as a loop, alternating with the economy loop.

- If `PLAYTEST.md` does not exist yet, this is the first pass: produce the playtest bot from scratch based on `problem.md`.
- If `PLAYTEST.md` already exists, it carries the exploit-investigator's `<review>` block. Read that feedback. Then open `MACRO.md` and check whether the latest economy model introduces bounds, invariants, or progression claims your current bot doesn't yet exploit or respect — for example a tighter generation limit your bot is now far from achieving, or an inflation cap your output violates. Revise `PLAYTEST.md` in place to address every reviewer point and to align with the latest economy limits. Don't discard the previous bot script and start over, and don't leave a flagged issue unaddressed.
