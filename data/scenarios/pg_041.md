# background

I need you to set up a 3-agent pipeline for quantitative trading alpha discovery: strategy_allocator, quant_researcher, and risk_manager.

- **strategy_allocator** orchestrates: each round it selects which idea to pursue next, calls the quant_researcher on that idea, extracts the report file path from the quant_researcher's reply, then calls the risk_manager on that path, decrements a counter, and loops. Does no analysis itself.
- **quant_researcher** explores the alternative datasets for the idea it is given, writes a signal report file, and records any new alpha opportunities it notices along the way.
- **risk_manager** evaluates the report against backtesting criteria and persists the verdict.

# f1: problem definition

You are participating in a quantitative trading alpha discovery competition.

Read the competition rules first: ./competition-rules.md

# f2: research tactics

These markets have been heavily traded for many years — any obvious arbitrage would have been exploited long ago. So:

- Don't report signals that are obvious at a glance (e.g. "simple moving average crossover") — if it were that obvious, large funds would have traded it away
- Truly valuable alpha hides in non-obvious alternative data interactions, structural market inefficiencies, and boundary conditions

**Writing the report**

Each idea corresponds to its own report file at `reports/{id}-{slug}.md`, where `{id}` and `{slug}` come from the idea's id and slug attributes in ideas.xml.

The report must contain four h2 sections in this fixed order:

- `## Signal Title` — concise, clear name for the alpha strategy
- `## Expected Performance` — what returns it can generate (Sharpe ratio, max drawdown, capacity, etc.)
- `## Alpha Hypothesis` — short writeup of the financial logic, with dataset/feature references
- `## Backtest Details` — a runnable Python script the risk_manager can execute as-is to backtest the signal

Do not modify or delete the `## Review` section — the risk_manager overwrites that themselves.

If after investigation you find the idea doesn't hold out of sample, still briefly record your analysis and the reason for rejection in the report file.

# f3: evaluation criteria

Check the following dimensions one by one:

| Dimension | Requirement |
|-----------|-------------|
| Scope | Within tradable liquid universes, or alternative datasets that directly map to liquid instruments |
| Data references | Cited datasets, column names, and API signatures must exactly match the latest live data environment |
| Expected performance | Claimed metrics consistent with the backtest's own output, no self-contradiction, no curve-fitting exaggeration |
| Financial rationale | Pinpoint specific market inefficiencies and explain why the alpha exists |
| Backtest script | End-to-end backtest script simulating live trading (must not use future data to simulate) |
| Reproducibility | Backtest runs end-to-end without manual assembly by the risk_manager; statistical arbitrage signals must backtest over 5 years of data in < 10 minutes |
| Deduplication | Search the internal factor library, production commits, and public academic literature to confirm it's not a known risk factor |
| Amplification | The signal's return must significantly exceed the baseline of "buying and holding the benchmark index" |

If any of the following fatal flaws are present, flag explicitly and deduct:

- References to nonexistent datasets or AI-hallucinated API signatures
- Performance claims contradict the backtest's own output
- Backtest script simulates trading by using look-ahead bias (future data)
- Data references inconsistent with the latest live data environment
- Backtest script is incomplete and requires the risk_manager to manually assemble it
- Strategy return is equivalent to buying and holding the benchmark index, with no pure alpha

# f4: allocation rules

You drive the explore → write → review → improve loop: each round, select the next idea, then alternate calls to the quant_researcher and risk_manager agents.

Read the remaining-round count from `counter.txt`. If the file doesn't exist or the count ≤ 0, print "rounds exhausted" and stop.

Loop:

```
while counter > 0:
    1. Select the next idea, then call the quant_researcher agent with it.
    2. Parse quant_researcher's reply and extract the report file path.
    3. If a report path is present, call the risk_manager agent with prompt: "Please review {path}".
    4. Decrement counter and write the new value back to counter.txt (whether or not the risk_manager was called).
    5. Loop back to step 1.
```

After the loop, summarize: how many rounds ran, how many distinct ideas were explored, and any anomalies encountered.

**Calling rules**

- Do **not** analyze datasets, write reports, or do reviews yourself — delegate everything to the two sub-agents.
- If quant_researcher or risk_manager errors out, append the error to `error-log.txt` (format: timestamp + agent name + error content), then continue with the next round.

Notes:
- quant_researcher and risk_manager each run in independent context per round; they do not pollute each other.
- All persistent state moves through files (`ideas.xml`, `reports/`, `counter.txt`).
- You don't need to understand any alpha signals — just pass file paths correctly.

# f7: idea selection

Each round, before calling the quant_researcher, read ideas.xml and select which idea the quant_researcher should work on:

1. Every idea carries an `attempts` count and a risk_manager `score` (0-10). `score="0"` means the idea is dead — never select it again. `score="none"` means it has not been attempted yet.
2. Prefer ideas with `attempts="0"` and `score="none"` first.
3. Otherwise rank by UCB value: `UCB = score + 10 × √(ln(N) / attempts)`, where N is the sum of all attempts; a higher score with fewer attempts ranks first.
4. Pass the selected idea to the quant_researcher. If ideas.xml holds no workable idea, call the quant_researcher with none and let it explore and surface fresh ones.

# f8: recording new ideas

While exploring the alternative financial datasets and market data, the moment you spot any potential new alpha signal — even if you are unsure it holds up to backtesting — immediately append it to ideas.xml (`score="none"`, `attempts="0"`). Don't wait until exploration is finished to batch-record, and don't judge feasibility before recording. Recording an idea is not a commitment to pursue it; it just keeps the lead from being forgotten. Whether it ever gets worked on is decided later, when the next idea is selected.
