# background

I need you to set up a 6-agent pipeline that solves mathematical optimization problems. A problem arrives as a natural-language statement in `problem.md`, and the pipeline turns it into a vetted plan and then working, verified solver code.

The six agents:

- **dispatcher** orchestrates the run: it drives the plan phase, then the code phase, and tracks progress. It does no modeling, coding, or reviewing itself.
- **plan-creator** is invoked once at init and produces K differentiated candidate plans, one per `plans/{id}-{slug}/PLAN.md`. It never enters the per-plan revision loop.
- **plan-refiner** is invoked once per plan-refinement step in the loop. It revises a single assigned plan's `PLAN.md` based on the plan-critic's last review for that plan. It never creates new plans.
- **plan-critic** reviews the plan it is currently assigned to.
- **coder** implements the strategy as solver code for the plan it is currently assigned to.
- **code-critic** reviews the solver code and result of the plan it is currently assigned to.

The pipeline hands work off through plan-namespaced files:

- `problem.md` — the original problem, given as input. All five domain agents (plan-creator, plan-refiner, plan-critic, coder, code-critic) read it.
- `plans/{id}-{slug}/PLAN.md` — one plan per directory, K plans total. The plan-creator writes K differentiated candidates at init; the plan-refiner revises a specific assigned plan in the loop; the plan-critic reviews the plan it is assigned to; the coder reads the plan it is assigned to.
- `plans/{id}-{slug}/SOLUTION.md` — the solver code and results for that plan. The coder writes it; the code-critic reviews it.

Each critic records its score as a `<review score=X>` block inside the file it reviewed, and the dispatcher reads those scores to drive the loops.

The dispatcher round-robins across the K plans, first in a plan phase (each plan iterates its plan-refiner ↔ plan-critic loop) and then in a code phase (each plan iterates its coder ↔ code-critic loop). Cross-plan reading does not happen: an agent assigned to one plan never reads another plan's files.

# f1: problem context

You are working on a mathematical optimization problem — it may be linear, mixed-integer, nonlinear, or a combination. The original problem is given as a natural-language statement in `problem.md`; read it before starting your part.

# f2: plan-creator's deliverable

Produce K differentiated candidate plans, each in its own `plans/{id}-{slug}/PLAN.md`. The K plans should differ meaningfully in their solving strategy so distinct routes can be compared downstream, not in trivial ways like variable renaming.

A complete plan in any `PLAN.md` has two parts — a formalized model and one chosen solving strategy.

The **model** must state:

- **Problem type** — LP, MILP, NLP, MINLP, QP, and so on; this must be exact.
- **Data** — the key numeric inputs and what each one represents.
- **Decision variables** — their types and domains, with descriptive names.
- **Objective function** — the exact expression to optimize.
- **Constraints** — every constraint, listed explicitly.

The **strategy** — weigh the options and commit to the single most promising route — must state:

- which solver to use and how to call it;
- the algorithm and its key initial settings;
- the reformulation or trick the route relies on;
- the biggest risk on this route.

# f3: solver toolbox

Which solver suits which kind of problem — use it to choose one for the strategy:

- **PuLP** — LP and MILP only; no nonlinear support.
- **Pyomo** — LP, MILP, NLP, MINLP; the broadest coverage, and solver-independent.
- **OR-Tools** — LP, MILP, constraint programming, and routing.
- **Gurobi** — top performance on MILP and convex/nonconvex QP; commercial license; not for general NLP.
- **SCIP** — the strongest open-source option for MINLP and global optimization; on pure MILP, slower than Gurobi.
- **IPOPT** — large-scale nonlinear problems.

Match the solver to the problem type; do not force an ill-suited one.

# f4: writing the code

Read the plan in your assigned plan's `PLAN.md` and implement its strategy as solver code:

- Include every import. Create each variable with the correct type and bounds, define the objective and every constraint exactly as the model states, solve, check the solution status, and print both the solution and the objective value.
- Run the code yourself and watch it through to completion — do not stop while it is still running. If it errors, find and fix the root cause; do not bury errors under try/except.
- Sanity-check the answer: a quantity that must be a whole number should not come back fractional.

Write the code and its results up in your assigned plan's `SOLUTION.md`. Do not read or touch other plans' files.

# f5: solution doc discipline

Keep your assigned plan's `SOLUTION.md` lean. Its body — everything except the `<review>` block — must stay under 3000 words (~20KB), about a four-page note; record only the final state and move iteration-by-iteration history into the plan's `CHANGELOG.md`. Bold text must not exceed 5% of the file: if everything is bold, nothing is.

# f6: plan review

Review the plan in your assigned plan's `PLAN.md`. Operate strictly on the assigned plan; do not look at other plans.

- **Model** — does it faithfully capture the problem? Check that the problem type, decision variables, objective, and constraints are all accurate.
- **Strategy** — is the chosen solver and algorithm a sound fit, is the route actually implementable, and is it likely to reach a correct solution?

Be brief and specific. Do not write any solver code yourself — only review.

# f7: code review

Review your assigned plan's `SOLUTION.md` — the solver code and its results. Verify it directly against the problem statement. Operate strictly on the assigned plan; do not look at other plans.

- Inspect the reported result for anomalies, unreasonable objective values, or constraint violations.
- Write a separate `verify.py` under the plan's directory that re-checks every constraint against the reported solution, and run it; name any variables that violate a constraint.
- Read the solver code and spot-check it for bugs — unused parameters, dead code, logic that contradicts what the solution says it does. A bug that silently degrades the solution is worse than one that crashes.
- Treat the result as finished only when every constraint holds and the objective value is sound — not merely when the code runs without error.
- Check the length of the plan's `SOLUTION.md`: if it exceeds 20KB, give a warning, but do not deduct points for it.
- Check the bold ratio of the plan's `SOLUTION.md`: if bold text exceeds 5% of the file, give a warning, but do not deduct points for it.

# f8: scoring

Score the work from 1 to 10, and record your verdict by overwriting the single `<review score=X>` block at the end of the document you reviewed — overwrite the existing one, never append a second.

Score strictly on real merit. Do not give any credit for a piece of work merely being honest about its own shortcomings — "this doesn't work, but I'm telling you it doesn't" is still a failing result, not a passing one.

# f9: orchestration

You orchestrate the run; you never model, code, or review yourself — you only dispatch the right agent at the right time.

- **Init** — dispatch the plan-creator once to draft K differentiated candidate plans, each in its own `plans/{id}-{slug}/` directory. After it returns, read the directory list to learn the K plan ids. The plan-creator is not invoked again after this.
- **Plan phase** — round-robin across the K plans. For each plan in turn, dispatch plan-refiner (revise that plan based on its plan-critic's review) then plan-critic (score that plan). After one round-robin pass through all K plans, start the next pass. A plan is considered done in this phase when its plan-critic score reaches 10; stop scheduling more plan-phase calls for it but continue with the others. After all K plans hit plan-critic 10, or each plan has had 10 plan-phase iterations, move to the code phase.
- **Code phase** — round-robin across the K plans. For each plan in turn, dispatch coder (revise that plan's solution based on its code-critic's review) then code-critic (score that plan's solution). The pipeline stops when any plan reaches code-critic score 10, or each plan has had 10 code-phase iterations.
- When dispatching, always pass the plan's directory path so the dispatched agent knows which `PLAN.md` / `SOLUTION.md` to operate on.
- Keep the current phase, per-plan iteration counts, per-plan latest scores, and current plan index in `STATE.md`, accurate after each dispatch. Read each score from the relevant plan's critic `<review score=X>` block. Commit progress after each step.

# f10: plan revision pass

Read your assigned plan's `PLAN.md`. It carries the plan-critic's `<review>` block. Read that feedback and revise the plan's `PLAN.md` in place to address every point raised. Preserve the two-part structure (formalized model plus chosen solving strategy). Don't discard the plan and start over, and don't leave a flagged issue unaddressed. You revise; you do not create new plans. Do not read or touch other plans' files.

# f11: code revision pass

Your solver code and its results live in your assigned plan's `SOLUTION.md`.

- If your assigned plan's `SOLUTION.md` does not exist yet, this is the first pass for that plan: produce the solution from scratch based on the plan's `PLAN.md`.
- If the plan's `SOLUTION.md` already exists, it carries the code-critic's `<review>` block. Read that feedback and revise the plan's `SOLUTION.md` in place to address every point raised. Don't discard the solution and start over, and don't leave a flagged issue unaddressed. Do not read or touch other plans' files.
